# v2.1.2 AI Harness Live Eval Report

Date: 2026-06-28

## Scope

This eval pass focused on the backend AI harness for the coding assistant and interpretation assistant.
It did not change the application UI/UX surface.

The goals were:

- Remove brittle answer-length control and replace it with stronger workflow-specific system prompts.
- Verify that the assistants stay concise, bounded, and reference-grounded.
- Catch incomplete responses, overconfident diagnostic/coding claims, unsafe treatment or legal framing, and weak HITL boundaries.
- Save enough process evidence for release notes and public showcase documentation without storing secrets or raw clinical-style responses.

## Safety And Privacy

- The OpenAI API key was loaded from a local secret file into `OPENAI_API_KEY` only for each command.
- Each PowerShell command removed `OPENAI_API_KEY` from the environment in `finally`.
- OpenAI Responses API calls use `store: false`.
- JSONL eval logs contain fixture IDs, status, issue types, token usage, output character counts, and estimated cost.
- JSONL eval logs do not contain API keys or raw model answers.
- Debug output was used only in the terminal during diagnosis and was not written to the repo logs.

## Harness Changes

- Added `lib/ai/harness.ts` as the central OpenAI harness layer for workflow profiles, response policies, output caps, metadata, and streaming completion summaries.
- Added workflow-specific response policies:
  - Coding assistant: candidate/provisional language, no automatic row edits, explicit clinician review.
  - Interpretation assistant: data anchors, provisional hypotheses, bounded diagnostic/treatment/legal scope, no invented variables.
- Hardened coding prompts for FQ, Popular/P, Z, M, and severe special scores such as FABCOM2.
- Hardened interpretation guardrails for diagnosis, treatment planning, legal/forensic risk, single-variable overreach, and broad first-pass questions.
- Added runtime retrieval contracts for coding-rule chunks and interpretation knowledge chunks.
- Added live eval support for OpenAI model, locale, fixture ID, retrieval mode, debug output, and model-specific output caps.

## Important Fixes Found By Eval

- The original eval was too brittle about exact wording. It failed safe answers such as "cannot make a legal judgment" because the phrase "legal judgment" appeared.
- The forbidden-phrase evaluator now allows clearly negated safety-boundary use while still failing unsafe affirmative claims.
- gpt-4o-mini sometimes expressed uncertainty without using exact words such as "candidate" or "hypothesis"; fixture contracts now accept equivalent safe uncertainty signals.
- The coding assistant needed a stronger Korean direct-action boundary: it now says it cannot edit rows automatically and can only discuss candidate codes for clinician review.
- Broad interpretation prompts needed a stronger instruction not to invent values, indices, or findings when no Structural Summary values are supplied.
- gpt-5.5 needed a higher live-eval output cap. The product profile already allowed higher output; the eval-only 700-token cap caused false `incomplete` failures.

## Current Limitations

- These evals are contract and regression tests, not a substitute for expert clinical review.
- The fixtures are synthetic and intentionally narrow; future releases should add real anonymized edge cases when available.
- The evals verify answer boundaries and retrieval behavior, but they do not yet grade clinical usefulness with a human rubric.
- Cost estimates use the local script price table and should be treated as engineering estimates, not billing records.
