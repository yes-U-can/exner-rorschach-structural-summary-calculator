# v2.2.0 Chat UX And Streaming Validation

Date: 2026-07-13 to 2026-07-14 (KST)

Status: local pre-release verification pass

## Scope

- Interpretation assistant and coding assistant chat behavior
- Independent message scrolling and user-controlled follow mode
- Response copy and feature-gated, anonymous helpful/not-helpful actions
- GPT-5.5 output-completion policy and live route evaluation
- User-controlled generation stop and end-to-end provider cancellation
- Exact continuation after an interrupted answer
- No scoring logic, reference corpus, embedding, or retrieval-ranking changes

## Defects Reproduced

### Document-level chat growth

The interpretation page used a flex item without a bounded height and contained nested `overflow-y-auto` elements. The message list therefore expanded the document instead of becoming the active scroll container.

### Forced and jittery auto-follow

Every streamed text update called `scrollIntoView()` on an end marker. That method could move every scrollable ancestor, including the document. Because the user was scrolling the page rather than the intended message element, the near-bottom check did not reliably observe user intent and auto-follow did not disengage.

### Confirmed output truncation

The detailed Korean interpretation route was reproduced with the production GPT-5.5, BYOK, prompt, RAG, and structured-stream path. The provider returned `status=incomplete`, `reason=max_output_tokens`, and exactly consumed the configured 2,200-token ceiling. The visible response stopped after 1,110 characters.

### UI status mixed into model context

The client appended the localized stream-interruption label directly to assistant content. That UI-only label could then be copied, persisted in session storage, and sent back as model conversation context.

### No user-controlled stop

Both assistants disabled the send button during generation and exposed no stop control. The browser therefore could not cancel response-body consumption or signal the server to stop the in-flight OpenAI request.

### Repetitive continuation after interruption

When a partial answer ended inside a numbered section, a follow-up such as "why did you stop" could restart that section and repeat its heading and evidence instead of continuing from the exact unfinished sentence.

## Implementation

### Scroll ownership

- Both assistants use the shared `useChatAutoScroll` hook.
- Streaming updates are coalesced to at most one container scroll per animation frame.
- The hook updates the message container directly and never uses document-level `scrollIntoView()`.
- Upward wheel, touch, scrollbar, and keyboard intent pauses follow mode.
- Returning near the bottom or pressing the down-arrow button resumes follow mode.
- `ResizeObserver` and viewport resize handling preserve the correct state through responsive text reflow.
- `overscroll-behavior: contain` prevents the message list from unintentionally chaining scroll to the page.

### Message actions

- Completed assistant messages always expose copy. Helpful and not-helpful buttons appear only after the dedicated feedback store is explicitly enabled.
- A rating is saved immediately, then an optional multi-select dialog offers predefined reasons. No free-form feedback field is rendered or accepted.
- Every icon button has a localized accessible name and tooltip in all five supported languages.
- Copy uses the modern Clipboard API with a compatibility fallback.
- Streaming-state notices are rendered separately from model-authored content.

### Privacy-minimized feedback

The browser converts exact response length into a coarse bucket before sending feedback. The endpoint rejects any payload containing unknown fields, including prompt text, response text, or an exact character count. The existing RAG database remains read-only and RAG-only. A separate feedback database stores only:

- random client feedback identifier
- helpful or not-helpful rating
- selected predefined reason codes and reason-schema version
- assistant workflow
- locale
- fixed model and harness identifiers
- completed, incomplete, or unknown status
- coarse response-length bucket
- timestamps

Feedback metadata expires after 180 days and is purged during later feedback writes. The implementation decision and rollback boundary are recorded in [`ADR-0003`](../adr/0003-privacy-minimized-ai-response-feedback.md).

It does not store prompts, responses, free-form comments, Structural Summary values, response memos, API keys, IP addresses, user agents, or user identifiers.

### Completion policy

- Harness protocol: `sicp-openai-harness-v3`
- Interpretation policy: `interpretation-balanced-continuity-v4`
- Coding policy: `coding-assist-balanced-continuity-v4`
- Interpretation ceiling: 8,000 tokens
- Coding ceiling: 6,000 tokens

The system prompt remains responsible for normal response length. The provider token cap is now an emergency ceiling rather than the primary brevity control. When a visible prior response ends mid-sentence or mid-section and the user asks to continue, the policy requires resuming at the exact unfinished point without restarting completed sections.

### Cancellation lifecycle

- During generation, the existing send control becomes a localized stop icon without changing the input-row dimensions.
- The browser aborts both `fetch()` and response-body consumption with `AbortController`.
- Next.js combines the request signal, a downstream response-cancellation signal, and the provider deadline.
- Downstream cancellation reaches the OpenAI SDK stream controller, stopping the in-flight network request instead of merely hiding output.
- User stop, provider timeout, output-length exhaustion, and unexpected connection loss have separate UI status messages.
- Partial model text remains available for reading and copying; status text stays outside model-authored content.

## Release Boundary

The dedicated feedback migration was validated only against an ephemeral local database. It has not been applied to a live database in this local pre-release batch. The buttons remain hidden by default. Deployment must provision a database that is separate from both RAG URLs, apply `prisma/feedback/migrations/20260713143000_init_ai_response_feedback`, and only then set `NEXT_PUBLIC_AI_FEEDBACK_ENABLED=1`.

The in-app test browser intentionally denied page clipboard access, so it exercised the visible failure state rather than the operating-system clipboard. The modern and compatibility copy paths are covered by dedicated unit tests; a final release smoke test should repeat the copy action in a normal production browser.
