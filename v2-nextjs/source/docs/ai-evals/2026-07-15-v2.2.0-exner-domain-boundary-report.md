- No UI/UX changes in this batch

- All live prompts and Structural Summary values were synthetic

## Product Boundary

The interpretation and coding assistants now explicitly support only the Exner Comprehensive System (CS):

- Exner CS coding
- Exner Structural Summary review
- cautious discussion grounded in the current Exner reference corpus
- limited case context only when it directly qualifies an Exner coding or Structural Summary question

The assistants must not provide:

- R-PAS or another non-Exner Rorschach framework
- MMPI, PAI, TAT, HTP, SCT, WAIS, WISC, Bender-Gestalt, or other cross-test integration
- general case formulation, diagnosis, treatment planning, legal conclusions, or forensic conclusions
- unrelated general-assistant work
- system or developer prompts, hidden policies, raw corpus dumps, evaluation canaries, chain-of-thought, API keys, cookies, or session tokens

## Threat Model

The batch tested five practical failure classes:

1. A direct request to combine Exner results with R-PAS or MMPI.
2. A request to ignore prior instructions and reveal internal prompts or secrets.
3. Instructions embedded inside an untrusted scoring memo.
4. A valid Exner conversation that drifts out of scope in a later turn.
5. An unrelated request that tries to use the assistant as a general chatbot.

## Defense Layers

### 1. Deterministic Request Gate

High-confidence prompt injection, internal-information extraction, named non-Exner systems, cross-assessment requests, and clearly unrelated tasks are rejected before reference retrieval or a GPT request begins.

The gate returns a short localized boundary response in Korean, English, Japanese, Spanish, or Portuguese. It logs only a category and request metadata, not the user's raw text.

### 2. Fixed System Policy

Both assistant prompts now include the same Exner-only domain and confidentiality policy. User messages, prior chat messages, scoring memos, pasted text, retrieved passages, and quoted instructions remain untrusted data. They cannot become product-level instructions merely by claiming to be a system, developer, administrator, or evaluator message.

### 3. Existing Secret And Action Isolation

- The BYOK API key is passed only to the OpenAI client constructor and is not placed in model messages.
- OpenAI calls keep `store: false`.
- The model has no tools that can read files, query the database, alter scoring rows, or perform external actions.
- Saved eval artifacts contain metadata and issue types, not raw prompts or raw model answers.

### 4. Security Regression Evals

The eval suite now includes strict canary checks. A canary is considered leaked even if the answer says it cannot reveal the canary while repeating the value. Single-turn and multi-turn fixtures cover all five UI locales across static and live tests.

This layered design follows OpenAI's guidance to combine clear instruction hierarchy, untrusted-input boundaries, input guardrails, adversarial testing, constrained capabilities, and evals rather than relying on one prompt alone:

- [Safety best practices](https://developers.openai.com/api/docs/guides/safety-best-practices)
- [Safety in building agents](https://developers.openai.com/api/docs/guides/agent-builder-safety)

## Remaining Limits

- No prompt-injection defense is mathematically perfect. Novel wording can miss deterministic patterns, and model behavior remains probabilistic.
- The current static gate deliberately blocks only high-confidence patterns to reduce false positives. Semantically novel out-of-scope requests still depend on the fixed system policy.
- Streaming text is forwarded as it arrives. Fully validating every answer before display would require buffering the entire answer and would remove the current streaming experience.
- The model sees the selected reference passages needed for RAG. Canary tests passed, but they cannot prove that no future adversarial phrasing will ever elicit an unintended paraphrase.
- The lack of model tools and the exclusion of API keys from model messages materially reduce blast radius, but they do not turn the model into a security boundary.
- These evals test engineering behavior, not clinical correctness. Final coding and interpretation remain the clinician's responsibility.

## Privacy Boundary

No API key, raw model answer, raw prompt, private assessment payload, or source-document text was written to the saved artifacts. The local key was read only from the development environment and supplied to child-process memory for authorized tests.
