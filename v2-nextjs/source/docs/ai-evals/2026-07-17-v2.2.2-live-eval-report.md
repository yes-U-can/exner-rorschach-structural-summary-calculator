## 먼저 바로잡는 이력

배포된 v2.2.1 계산 코드와 공개 릴리즈 노트는 Lower Section의 관례적 `FC:CF+C` 항목을 처음부터 `FC:(CF+C+Cn)`으로 일관되게 계산하고 설명했다. 릴리즈 뒤 내부 재검토 과정에서 Cn을 다시 제외하는 미커밋 초안이 만들어졌지만, Git 이력과 계산 근거를 대조한 독립 감사에서 잘못된 방향으로 확인되어 공개 전에 철회했다.

따라서 v2.2.2는 배포 계산값을 뒤집는 릴리즈가 아니다. 서로 다른 Cn 계산 경계를 별도 변수, 회귀 테스트, 5개 언어 지식 문서와 AI 평가 계약으로 더 분명하게 고정한 릴리즈다.

## 검증할 네 경계

- 화면의 관례적 `FC:CF+C` 우변은 `CF+C+Cn`
- WSumC는 Cn 제외
- S-CON의 `CF+C > FC`는 Cn 제외
- Color-Shading blend는 Cn 제외
- 수치 예시 `FC=2`, `CF=4`, `C=0`, `Cn=1`의 화면 결과는 `2:5`

계산식의 정답은 TypeScript 회귀 테스트와 독립 오라클에서 판정한다. live eval은 교정된 지식이 코퍼스, 벡터 검색, 프롬프트 조립, GPT-5.5 응답까지 전달되는지를 확인한다. GPT-5.5 답변 자체를 계산 규칙의 정답지로 사용하지 않는다.

## 무엇을 고쳤는가

### 코딩 도우미 production과 같은 검색·프롬프트 조립

코딩 도우미는 부호 문서만 검색해서는 계산 뒤쪽의 Lower Section 경계를 놓칠 수 있다. production 경로는 primary coding chunk에 연결된 supporting interpretation chunk도 함께 조립한다. 코딩 live harness도 같은 함수를 사용하도록 바꿔 이 workflow의 시험 경로와 실제 서비스 경로를 일치시켰다. 반면 해석 live harness는 검색 개수, supporting coding 지식 병합, 구조요약 CSV system block, lexical fallback이 `/api/chat`과 다르므로 production parity 근거가 아니라 제한된 smoke 근거로만 사용한다.

Cn 계산 경계를 명시적으로 묻는 질의는 5개 언어 모두 `scoring-input/determinants/Cn`의 완전한 경계 문서를 우선 anchor한다. 한국어와 일본어처럼 `Cn` 뒤에 조사가 바로 붙는 문장도 ASCII 코드 경계로 인식한다.

5개 언어 Cn 문서와 관련 Lower Section 문서에는 다음 내용을 같은 의미로 맞췄다.

- 화면 비율은 `FC:(CF+C+Cn)`
- WSumC, S-CON, Color-Shading은 Cn 제외

작성 원문에서 route 문서와 chunk 코퍼스를 다시 만들고 vector promotion을 실행했다. 최종 벡터 상태는 6,629/6,629, stale 0, OpenAI 5개 언어 ready다. 전 언어 prune이 locale을 무시하던 운영 결함도 언어별 chunk ID 비교로 교정하고 회귀 테스트를 추가했다.

### 사실은 엄격하게, 문체는 의미로 검사

평가 계약은 다음 원칙으로 정리했다.

1. `2:5`, `CF+C+Cn` 같은 수식형 신호는 문자 그대로 존재해야 한다.
2. `12:5`나 `2:55` 안의 부분문자열은 `2:5`로 인정하지 않는다.
3. WSumC, S-CON, Color-Shading과 Cn 배제 의미가 같은 문장 블록 안에 있는지 각각 검사한다.
4. 단어 순서나 자연스러운 번역 차이는 허용하지만 네 계산 경계 중 하나라도 빠지면 통과시키지 않는다.
5. `S-CON에서 사용하는 비율에는 Cn을 제외한다`는 올바른 설명을 `화면 비율에서 Cn을 제외한다`는 오류로 오인하지 않는다.
6. 금지 문구의 부정 여부는 해당 문구가 있는 문장 안에서만 판정해, 앞 문장의 부정어가 뒤 문장의 오답을 가리지 못하게 한다.
7. 일본어 무공백 문장과 240자를 넘는 장문도 고정 길이 자동 실패 없이 겹치는 의미 창으로 검사하며, `WSumCにCnを含めます` 같은 무공백 오답도 잡는다.

즉 기준을 느슨하게 만든 것이 아니다. 수식과 관계는 더 직접적으로 검사하고, 특정 모범 문장 암기는 요구하지 않도록 바꿨다.

## 보안과 개인정보

- 모든 fixture는 인공 문장이다.
- API key, raw prompt, raw answer, 반응 메모 원문, 구조요약 payload를 JSONL에 저장하지 않는다.
- 저장 값은 fixture ID, 모델, token 사용량, retrieval mode, hit 수, route, issue 유형, 완료 상태와 비용 추정치다.
- 사설 저장소의 JSONL은 재현 추적용 source commit을 가질 수 있지만, 공개 미러에서는 commit 계열 식별자를 재귀적으로 제거한다.
- 진단용 raw 출력은 일시적인 로컬 콘솔에서만 확인했고 저장 코드와 환경 플래그는 최종 작업본에서 제거했다.
