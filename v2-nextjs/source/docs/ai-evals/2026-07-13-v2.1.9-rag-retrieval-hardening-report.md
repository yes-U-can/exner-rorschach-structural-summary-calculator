# v2.1.9 RAG Retrieval Hardening Report

Status: released

Date: 2026-07-13

Product model: BYOK OpenAI assistant with clinician-in-the-loop boundaries

UI/UX change: none; release version metadata is updated separately

## What This Release Changed

v2.1.9 improved how the coding and interpretation assistants find reference documents before answering. It did not change the calculator UI/UX or expand the clinical scope of the product. The calculator remained available without an AI session, and the assistants continued to support rather than replace clinician judgment.

## Scope Completed In v2.1.9

This release corrected low-signal one-character matching, weak Korean and Japanese query handling, and a score-fusion formula that could let a noisy word match outrank a stronger meaning match. It replaced direct score addition with weighted Reciprocal Rank Fusion (RRF), removed duplicate routes, required a minimum meaning-match signal, improved five-language word handling, and limited coding-assistant search queries to explicitly selected evidence.

Four remaining problems were corrected in v2.1.10: broad fallback result/record alignment, Japanese text attached directly to Rorschach symbols, broader natural-language interpretation requests, and replaying all database changes on a new database. See the [v2.1.10 follow-up fix report](./2026-07-13-v2.1.10-final-independent-audit-closure-report.md).

The 365 checks reported for v2.1.9 consist of 360 prepared retrieval questions plus 5 data-shape checks. Japanese symbol preservation covered separated forms in v2.1.9; attached forms such as `FQ+の...` were completed in v2.1.10. The reported broad-query contamination rate of `0%` describes the development expressions used for this release, not every possible paraphrase.

## Implemented Changes

### Five-locale lexical retrieval

- Filters low-signal one-character natural-language tokens while preserving explicitly scoped Rorschach codes such as `Card I`, `Content A`, and lowercase determinant `m`.
- Preserves parenthesized structural-summary expressions such as `3r+(2)/R` as a complete retrieval term instead of over-ranking the internal `/R` fragment.
- Expands Korean particles and common verb endings before matching.
- Adds `Intl.Segmenter` word segmentation for Japanese while preserving separated symbols such as `DQ+`, `FQ+`, `M-`, and `v/+`. Attached Japanese suffix forms were completed in v2.1.10.
- Adds broad interpretation intent in all five locales and section cues that disambiguate duplicate symbols such as `a:p` and `Zf`.

### Hybrid fusion

- Replaces incomparable lexical-score and cosine-score addition with weighted RRF using `k=60`.
- Uses separate conservative weights for interpretation and coding retrieval.
- Rejects vector candidates below cosine similarity `0.32`.
- Deduplicates candidates by canonical route before fusion.
- Applies lexical-vector agreement bonus once per route.
- Keeps exact-route and metadata signals as bounded rerank bonuses. A stacked exact match can exceed one single-source RRF contribution, but does not restore direct lexical-score domination.
- Constrains a clearly broad interpretation request to interpretation routes and preserves the interpretation root anchor.

### Coding-assistant context minimization

- Builds the embedding query from the focus row and explicitly selected supporting rows only.
- Excludes unrelated sheet-wide rows.
- Caps the query at 12 rows, 800 characters per memo, and 3,000 characters total.
- Keeps full generation context and UI behavior unchanged; only the retrieval query is minimized.

### Vector observability and cleanup

- Serializes vector refresh timestamps as UTC and compares corpus timestamps with an explicit UTC-safe PostgreSQL expression.
- Reports providers outside the OpenAI-only runtime policy instead of silently omitting them.
- Adds an OpenAI-only Prisma migration that removes legacy Google embedding rows and replaces the provider enum.
- Verifies the destructive migration in an isolated temporary pgvector/PostgreSQL database: Google rows were removed, OpenAI rows were preserved, and the enum contained only `openai`.
- Makes validation fail when unsupported provider rows remain. The production migration removed 6,597 legacy Google rows, preserved all 6,632 OpenAI rows, reduced the enum to `openai`, and produced a snapshot with `providerAudit.clean: true`.

No OpenAI re-embedding was performed because all 6,632 OpenAI vectors are current for the corpus and report stale count `0`.

## Privacy And HITL Boundary

The public artifacts contain fixture identifiers, locales, routes, ranks, similarity or aggregate score data, completion status, issue types, token usage, output length, and estimated cost where available.

They do not contain:

- API keys or environment files
- raw prompts, chat messages, or model answers
- private assessment payloads or patient data
- private source documents or private provenance
- local runtime logs, caches, or Vercel state

The release changes retrieval and evaluation behavior. It does not allow the assistant to assign final codes automatically, issue diagnoses, replace clinician judgment, or bypass the existing HITL boundaries.

## Remaining Limits

- The 25-query hybrid set was used during development and is not blind held-out evidence. A genuinely frozen set authored and reviewed outside the tuning loop is still needed.
- The 360 curated retrieval queries and 5 schema/contract checks are strong regression contracts but remain alias-heavy and cannot estimate real-world retrieval accuracy by themselves.
- Automated phrase and rubric contracts test engineering behavior, not clinical correctness.
- The production-like route eval currently covers one synthetic interpretation scenario. More route-level coding, multilingual, error, and interrupted-stream scenarios are still desirable.
- GPT-5.5 is stochastic. Clean final-pass runs do not guarantee every future response.
- Expert human review remains necessary for clinical usefulness, multilingual nuance, and the safety of real assessment interpretation.
