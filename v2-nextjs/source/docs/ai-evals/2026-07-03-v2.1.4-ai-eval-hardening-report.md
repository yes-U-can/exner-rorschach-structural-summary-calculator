# v2.1.4 AI Eval Hardening Report

Date: 2026-07-03

Status: released and deployed as v2.1.4

## Goal

v2.1.4 focuses on AI answer-quality evaluation, not UI/UX.

The goal is to make the OpenAI assistant harness easier to trust before release:

- test follow-up behavior across multiple turns
- preserve human-in-the-loop boundaries across row changes and repeated questions
- verify that human review scores match the written rubric
- keep saved eval evidence privacy-safe and publishable

## Baseline Before This Batch

The app already had:

- single-turn AI harness fixtures
- single-turn contract checks for missing signals, forbidden overclaims, empty output, and likely truncation
- retrieval contract tests
- JSONL artifact auditing
- a draft human review rubric
- single-turn live OpenAI eval entry points

The missing layer was reproducible multi-turn evaluation and machine-checkable human review records.

## What Changed

### Multi-Turn Contract Evals

Added multi-turn fixtures for:

- coding-assist FQ follow-up after row change
- coding-assist movement follow-up after user asks to set a code
- interpretation low-R follow-up with diagnostic pressure
- interpretation missing-data follow-up with user pressure to assume values

Added a transcript evaluator that runs the existing answer contract on every assistant turn and also fails turn-count mismatches.

### Human Review Record Scoring

Added a JSONL scorer for human review records.

The scorer checks:

- required rubric fields
- valid workflow-specific dimensions
- integer dimension scores from `0` to `4`
- weighted score math
- blocking failures forcing `fail`
- optional release gate requiring every reviewed record to pass
- absence of key-shaped secrets and raw prompt/answer/message fields

### Live Multi-Turn Entry Point

Added a separate live OpenAI multi-turn Vitest suite and wired the batch runner to choose either:

- `--suite single`
- `--suite multiturn`

The explicit batch runner loads `.env.local` and `.env.production.local` before checking `OPENAI_API_KEY`. Normal Vitest live suites still skip unless the key is already present in the current process, so ordinary test runs do not accidentally spend API budget.

## Remaining Gaps

- The current human review records are synthetic examples. They prove the scorer works, but they are not clinical review of fresh model answers.
- A real v2.1.4 release candidate should run a multi-turn live batch and then create human review records from a sampled final-pass artifact.
- The scorer validates rubric math and privacy shape. It does not judge clinical quality by itself; that still requires a human reviewer.
