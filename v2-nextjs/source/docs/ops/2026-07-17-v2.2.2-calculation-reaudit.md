# v2.2.2 구조요약 계산 정확성 검산 보고서

> **2026-07-23 추가 정정:** 이 보고서가 all-F Lambda를 `∞`로 유지한 판정은 이후 재검토에서 뒤집혔습니다. 이 앱은 `∞`를 임상 출력값으로 표시하지 않고, 모든 반응이 순수 F인 기록에서 Lambda를 순수 F의 개수로 보고하는 소프트웨어 관행을 채택했습니다. 최신 판정과 근거 상태는 [구조요약 계산 규칙 권위 근거 재검토](./2026-07-23-authoritative-calculation-rule-audit.md)를 기준으로 봅니다.

검산 완료일: 2026-07-17 (KST)<br>
릴리즈: v2.2.2<br>
대상: 이 웹앱이 실제로 표시하는 Exner Comprehensive System 구조요약의 **Upper Section, Lower Section, Special Indices**

## 먼저 읽을 결론

v2.2.2는 새로운 구조요약 체계나 해석 규칙을 덧붙인 버전이 아닙니다. v2.2.1에서 바로잡은 계산을 다시 확인하고, Cn이 들어가는 계산과 들어가지 않는 계산을 서로 분리한 정확성 핫픽스입니다.

**완성된 프로토콜에서 v2.2.1의 `FC:CF+C` 오른쪽 값은 이미 Cn을 포함해 올바르게 계산됐습니다. 이 이유만으로 다시 계산할 필요는 없습니다.** v2.2.2에서 계산 결과가 새로 달라질 수 있는 부분은 형태질(FQ)을 입력하지 않은 미완성 행입니다. 이런 행은 FQ를 입력하기 전까지 GHR 또는 PHR로 임시 분류하지 않도록 고쳤습니다.

가장 중요한 계산 기준은 다음과 같습니다.

1. 화면의 관례적 항목명 `FC:CF+C`는 그대로 유지하지만, 이 앱이 기준으로 삼은 구조요약에서 화면 우변은 `CF+C+Cn`입니다.
2. `WSumC`는 `0.5*FC + 1.0*CF + 1.5*C`이며 Cn을 포함하지 않습니다.
3. S-CON의 `CF+C > FC` 판정은 표기 그대로 CF와 C만 비교하며 Cn을 포함하지 않습니다.
4. Color-Shading blend를 셀 때의 chromatic color 범위도 FC, CF, C이며 Cn을 포함하지 않습니다.
5. 이름이 비슷해도 위 네 계산은 서로 다른 목적과 경계를 가집니다. 하나의 공용 색채 합계를 만들어 재사용하면 오류가 생깁니다.

현재 계산 코드는 이 네 경계를 각각 다른 값으로 계산합니다. 같은 문제가 다시 생기지 않도록 각 계산식과 대표 예시를 자동 검사에 따로 넣었고, AI 도우미의 다섯 언어 설명도 같은 기준으로 맞췄습니다.

## 1. 검산 범위와 제품 경계

### 포함한 범위

- 현재 계산기가 화면에 표시하는 Upper Section 집계
- 현재 계산기가 화면에 표시하는 Lower Section 계산과 비율
- PTI, DEPI, CDI, S-CON, HVI, OBS의 조건 판정과 표시
- 위 결과를 설명하는 5개 언어 AI 참조 문서와 운영형 변수 설명
- 계산기와 AI 도우미에서 같은 오류가 다시 생기지 않는지 확인하는 자동 검사와 실제 GPT-5.5 답변 검사
- 공개 저장소의 계산 계보, 출처 역할, 수정 이력 설명

### 포함하지 않은 범위

- R-PAS 또는 다른 로샤 체계
- 이 앱 구조요약에 없는 연구용·확장 지표
- 인간 전문가가 원반응과 Inquiry를 보고 어떤 부호를 부여할지에 대한 자동 판정
- 구조요약 수치로 임상 진단이나 최종 해석을 자동 확정하는 기능
- 연령 입력 UI 또는 새로운 개인정보 필드
- 모바일 UI/UX 최적화

API 키 없이 사용하는 이 제품의 본체는 **입력된 부호를 계산하는 구조요약 계산기**입니다. 연령이나 면담정보처럼 해석에만 필요한 정보는 계산기 입력으로 강제하지 않습니다. AI 도우미를 선택해 사용한 대화에서 특정 해석에 연령이 꼭 필요할 때만, AI가 그 이유를 설명하고 대화 안에서 질문합니다.

## 2. 무엇을 정답 기준으로 삼았는가

이번 검산은 프로그램 하나를 정답지로 삼지 않았습니다. 서로 충돌할 때 다음 순서로 판단했습니다.

1. 이 앱이 기준으로 삼은 Exner Comprehensive System 판본의 명시적 계산식과 순서 규칙
2. 같은 판본에 실린 완성 구조요약 사례와 점수열
3. 서로 독립적으로 제작된 공개 계산 프로그램의 실제 수식과 출력 사례
4. 점수열과 완성 구조요약이 함께 공개된 사례
5. 2019 Excel, RorScore, v1 GAS, v2 TypeScript의 실제 코드 계보
6. 경계값 예시, 별도로 작성한 계산식, 같은 조건으로 반복 가능한 가상 자료, 언어가 바뀌어도 같은 부호에서 같은 값이 나오는지 보는 검사

여기서 중요한 구분은 **규칙 근거**, **독립 구현 증거**, **개발 계보 자료**가 서로 같은 역할을 하지 않는다는 점입니다.

- Exner 판본의 명시 규칙과 완성 사례는 계산 규칙을 결정하는 1차 기준입니다.
- CHESSSS, RAP3, RIAP, RorScore는 서로 별도로 만들어진 프로그램 또는 출력 자료입니다. 서로 일치하면 교차검증에 도움이 되지만, 어느 하나만으로 규칙을 확정하지 않았습니다. CHESSSS는 공개 설명서와 배포 페이지만 다시 확인할 수 있었으므로, 배포 파일의 셀 수식은 수식 단위 근거로 사용하지 않았습니다.
- 2019 Excel과 RorScore는 이 프로젝트가 어떻게 시작되었는지를 설명하는 계보 자료이기도 합니다. 이 자료에 구현된 수식을 그대로 권위 규칙으로 승격하지 않습니다.
- LLM 답변은 계산 정답지가 아닙니다. GPT-5.5 호출은 교정된 지식이 실제 검색·응답 경로에서 유지되는지만 확인합니다.

## 3. 개발 재료가 실제 코드에 미친 영향

### 3.1 2019 Excel workbook

v1 개발 당시 참고한 workbook은 [공개 배포 게시물](https://blog.naver.com/jin_k84/221539279596)에서 내려받을 수 있었던 계산 도구입니다. workbook 내부에서 확인되는 제작 표기는 `[Scoring Program] _by. Ju-Ri`까지이며, 이를 넘어 실명을 추정하지 않습니다.

이 workbook은 구조요약의 셀 배치와 많은 기본 합계를 프로그램으로 옮기는 데 실질적인 출발점이 되었습니다. 그러나 전체가 하나의 정확도 수준을 가진 것은 아니었습니다.

맞았던 대표 항목:

- `WSumC = 0.5*FC + CF + 1.5*C`
- EA, es, AdjEs의 기본 합산
- 자기중심성 지수에서 반사반응을 3배 가중하는 구조
- 화면의 `FC:CF+C` 우변에 Cn을 포함하는 셀 수식

달랐던 대표 항목:

- D와 AdjD를 -5에서 +5 사이로 제한
- EB 양쪽이 양수이면 공식 gate와 무관하게 EBPer를 표시
- WDA%와 Afr의 분모가 0일 때 spreadsheet 오류 발생
- S-CON과 Color-Shading 계산에 화면 표시용 Cn 포함 합계를 재사용

재현 가능한 셀 계보는 다음과 같습니다.

- 결정인 원계수는 `Upper_Section!AF8:AF11`의 `FC`, `CF`, `C`, `Cn` 라벨과 `AH8:AH11`의 빈도로 구성됩니다.
- 화면 비율은 `Lower_Section!Q9 = Upper_Section!AH8`과 `Lower_Section!S9 = SUM(Upper_Section!AH9, Upper_Section!AH10, Upper_Section!AH11)`입니다. 따라서 우변은 `CF+C+Cn`입니다.
- `Lower_Section!S11 = SUM(Upper_Section!AH8*0.5, Upper_Section!AH9, Upper_Section!AH10*1.5)`이므로 WSumC는 Cn을 포함하지 않습니다.
- S-CON criterion 7은 `Special_Indices!D15 = IF(Lower_Section!S9 > Lower_Section!Q9, "v", "")`입니다. 화면용 `S9`를 그대로 재사용하므로 Cn이 criterion 7에 섞입니다.
- Color-Shading은 `Scoring!AF5`가 `FC`, `CF`, `C`, `Cn` 중 하나를 Color로 표시하고, `AG5`가 shading 결정인을 표시한 뒤, `AH5`가 둘의 결합을 `C-S Blend`로 기록합니다. `AH4 = COUNTIF(AH5:AH104, "C-S Blend")`이고 `Special_Indices!D10`이 이 합계가 0보다 큰지 검사하므로, Cn과 shading이 함께 있는 반응도 포함됩니다.

따라서 “Excel이 전부 틀렸다”도 아니고 “Excel과 같으니 앱이 맞다”도 아닙니다. 항목별로 규칙과 수식을 분리해서 검증해야 합니다.

### 3.2 RorScore Perl 구현

초기 개발 과정에서 참고한 Perl 코드는 현재 공개된 [RorScore 저장소](https://github.com/jleader/RorScore)의 [`bin/rorschach.pl`](https://github.com/jleader/RorScore/blob/master/bin/rorschach.pl)과 본문이 일치하는 구현임을 다시 확인했습니다. 공개 링크는 이 원본 저장소를 기준으로 적었습니다.

검토한 canonical RorScore 구현에서 확인한 도움:

- 구조요약 입력을 코드와 배열로 표현하는 방법을 보여 주었습니다.
- EBPer의 EA, Lambda, 차이 gate는 규칙에 가깝게 구현되어 있었습니다.
- 자기중심성 지수의 반사반응 3배 가중과 ZEst 표도 유용한 비교점이었습니다.

검토한 canonical RorScore 구현을 그대로 따를 수 없었던 부분:

- 화면의 `FC:CF+C` 우변은 Cn을 제외합니다.
- D 변환표의 지원 범위를 벗어나면 연속 확장하지 않고 중단합니다.
- GHR/PHR을 산출하지 않고 입력된 값을 집계합니다.

v1의 모든 오류를 RorScore 탓으로 돌릴 수는 없습니다. 예를 들어 v1의 단순 EBPer는 RorScore보다 2019 Excel과 수작업 이식의 영향을 더 강하게 받은 형태입니다. 반대로 화면 비율에서 Cn을 누락한 구현은 RorScore의 단순화된 수식과 같은 방향이지만, 정확한 원인을 단일 자료로 단정할 기록은 없습니다.

### 3.3 v1 GAS의 수작업 이식

v1은 Excel 셀 수식과 Perl 구현을 사람이 읽고 Google Apps Script로 다시 표현한 프로젝트였습니다. 이 과정에서 자료 자체의 차이와 수작업 변환 오류가 섞였습니다.

- D/AdjD의 +/-5 제한은 Excel 동작을 계승했습니다.
- EBPer의 단순 비율은 Excel 동작을 계승했습니다.
- 화면의 Cn 누락은 Excel의 올바른 셀 수식을 놓치고 단순 `CF+C` 계열로 구현한 차이입니다.
- GHR/PHR은 RorScore에서 물려받은 것이 아니라 v1에서 별도로 구현했으며, 초기에는 7단계 우선순위 일부가 축약되었습니다.
- 자기중심성 지수의 반사반응 x2는 원자료가 아니라 수작업 이식 과정에서 생긴 오류입니다.
- ZEst 최종 경계의 off-by-one도 수작업 구현 오류였습니다.

이 구분이 중요한 이유는 수정 방법이 다르기 때문입니다. 원자료의 잘못은 더 높은 우선순위 규칙과 대조해야 하고, 이식 오류는 원자료와 코드 사이의 계보 비교로 잡아야 합니다.

### 4.2 Excel 오류는 임상적으로 얼마나 특수한 경우에 나타나는가

공개된 발생률 자료가 없으므로 임의의 퍼센트를 만들지 않습니다. 대신 오류가 실제로 켜지는 **정확한 발동 조건**을 제시합니다.

| 오류 | 발동 조건 | 범위 판단 |
| --- | --- | --- |
| D/AdjD 상한 | `EA-es` 또는 `EA-AdjEs`가 +15보다 크거나 -15보다 작음 | 큰 자원-부담 불균형이 필요한 극단 구간 |
| EBPer 과잉 표시 | EB 양쪽이 양수지만 EA, Lambda 또는 구간별 차이 gate 중 하나를 충족하지 않음 | 희귀 부호 없이도 가능해 비교적 넓음 |
| S-CON Cn 혼입 | Cn이 있고 `CF+C`와 FC가 판정 경계에 가까움 | Cn이 드물어 좁지만 S-CON 1점이 달라질 수 있음 |
| Color-Shading Cn 혼입 | Cn과 shading 결정인이 한 반응에 같이 있고 blend 문턱에 걸림 | 매우 특수한 조합 |
| WDA% 0분모 | W, WS, D, DS 반응이 전혀 없음 | 표준 실시에서 매우 이례적 |
| Afr 0분모 | I-VII 카드 반응이 전혀 없음 | 불완전 입력에 가까운 이례적 상태 |

D/AdjD 예시: 프로토콜 합계가 `EA=3.5`, `es=19.0`이면 원차는 `-15.5`입니다. Excel은 `D=-5`에서 멈추지만 연속 2.5 간격 변환은 `D=-6`입니다. 이는 한 반응 문장만으로 생기지 않고 `FM`, `m`, `Y`, `C'`, `T`, `V` 계열이 프로토콜 전체에 누적될 때 나타납니다.

기전 설명용 가상 응답은 “어두운 구름이 바람에 밀려 내려오고 있어요”입니다. Inquiry에서 무생물 움직임과 diffuse shading 사용이 실제로 확인되면 `m`과 `Y` 계열 후보가 될 수 있습니다. 이 문장 하나만으로 부호를 확정하거나 D=-6을 만든다는 뜻은 아닙니다.

EBPer 예시: `M=7`, `WSumC=4`, `EA=11`, `Lambda=.55`이고 양쪽 차이가 3이면 EBPer는 `1.75`입니다. v2.2.0은 비율 자체가 2.5 이상이어야 한다는 잘못된 조건 때문에 이 값을 숨겼습니다. 이 오류는 Cn 같은 희귀 부호가 없어도 발생하므로 실제 적용 범위가 상대적으로 넓습니다.

Cn 예시: “이건 빨간색이에요. 모양은 모르겠고 그냥 빨간색입니다.”가 Inquiry에서 순수한 색 이름 반응으로 확인되면 `Cn`, FQ none 후보가 됩니다. `FC=2`, `CF=4`, `C=0`, `Cn=1`이라면:

- 화면의 관례적 `FC:CF+C` 항목은 `2:5`로 표시합니다.
- WSumC는 `0.5*2 + 1.0*4 + 1.5*0 = 5.0`이며 Cn을 더하지 않습니다.
- S-CON은 `CF+C > FC`, 즉 `4 > 2`를 판정합니다. 화면 우변 5를 재사용하지 않습니다.
- Color-Shading blend에서도 Cn은 chromatic color로 세지 않습니다.

### 4.4 앱 오류는 임상적으로 얼마나 특수한 경우에 나타나는가

- **D/AdjD 제한:** 원차가 +/-15를 벗어나는 좁고 극단적인 구간입니다. 방향은 같아도 심한 정도를 한 단계 이상 축소했습니다.
- **EBPer:** 희귀 부호가 필요하지 않아 결함 중 적용 범위가 가장 넓습니다. 공식 완성 예제와 같은 일반적인 숫자 조합에서도 값이 누락될 수 있었습니다.
- **GHR/PHR:** 좋은 FQ의 인간표상에 `ALOG`, `CONTAM`, 또는 Level 2 인지 특수점수가 같이 있을 때처럼 흔하지 않지만 명시적으로 가능한 조합에서 우선순위가 달라질 수 있었습니다.
- **화면 Cn 누락:** Cn이 하나 이상일 때만 나타나므로 비교적 좁습니다. 영향은 Lower Section 화면 비율 우변입니다.
- **S-CON 또는 Color-Shading Cn 혼입:** Cn과 해당 판정 경계가 동시에 있어야 하므로 더 좁습니다. 하지만 S-CON은 총점 경계에서 한 항목 차이가 표시를 바꿀 수 있어 별도 식이 필요합니다.
- **WDA%/Afr 0분모:** 표준 실시에서는 매우 이례적입니다. 중요한 것은 정의할 수 없는 값을 정상적인 0으로 오인하지 않는 것입니다.

GHR/PHR 기전 예시: “두 사람이 서로 밀고 있어요. 둘이 떨어져 있으니 같은 사람인 게 분명해요.”라는 응답에서 영역, 형태, 운동이 확인되고 마지막 설명이 부적절한 논리로 판정된다면 `Do Ma H ALOG` 같은 후보가 될 수 있습니다. 좋은 FQ의 인간표상이어도 ALOG가 있으면 7단계 중 먼저 적용되는 Step 2에서 PHR입니다. 최종 부호화는 원반응과 Inquiry를 본 인간 채점자가 수행합니다.

### 4.5 무엇을 기준으로 맞고 틀림을 판단했는가

1. 명시된 산식을 TypeScript 조건식으로 별도 전사했습니다.
2. 완성 구조요약 사례는 입력 점수열부터 결과까지 end-to-end로 다시 계산했습니다.
3. CHESSSS 1의 공개 설명서와 배포 페이지를 확인했습니다. 배포 파일의 셀 수식은 같은 공개 경로에서 다시 확보하지 못했으므로, 직접 확인할 수 있는 최종 수식 근거에서는 제외했습니다.
4. RorScore, 2019 Excel, v1 GAS, v2 코드를 항목별 계보표로 비교했습니다.
5. D/AdjD, EBPer, GHR/PHR, 분모 0, Cn 경계를 최소 반례로 분리했습니다.
6. 구현과 같은 식을 복사한 테스트만 믿지 않고, 별도 오라클과 공개 완성 사례를 함께 사용했습니다.
7. GPT-5.5는 계산식 판정자가 아니라 교정된 지식을 검색하고 설명하는 경로의 검증 대상으로만 사용했습니다.

## 5. Cn이 특히 혼동을 일으킨 이유

구조요약 화면의 관례적 라벨은 `FC:CF+C`입니다. 라벨만 문자 그대로 읽으면 우변을 CF와 C의 합으로 구현하기 쉽습니다. 실제로 공개 구현 사이에서도 차이가 있었습니다.

- 2019 Excel의 화면 비율 셀은 Cn을 포함합니다.
- 앱이 기준으로 삼은 판본의 명시 규칙과 Cn>0 완성 사례는 Cn 포함을 지지합니다.
- [RAP3 구조요약 문서](https://virtualpsychology.com/documentation/rap_plus/Structural%20Summary%20report%20-%20Virtual%20Psychology.pdf)는 Lower Section 비율을 `FC:CF+C+Cn`으로 풀어 씁니다. 1쪽 `Single` 열은 단일 결정인만 세어 `FC=2`, `CF=0`, `C=0`, `Cn=1`로 표시합니다. 별도 `Blends` 열에는 `CF`가 포함된 blend가 2개, `C`가 포함된 blend가 1개 있으므로 전체 빈도는 `FC=2`, `CF=2`, `C=1`, `Cn=1`입니다. 이 전체 빈도로 2쪽의 화면 비율 `2:4`와 `WSumC=0.5*2+1.0*2+1.5*1=4.5`가 함께 성립합니다. 즉 화면 비율에는 Cn을 포함하지만 WSumC에는 포함하지 않는 경계를 실제 출력에서 확인할 수 있습니다.
- 검토한 canonical RorScore 구현은 라벨을 문자 그대로 구현해 Cn을 제외합니다.
- [RIAP5 공개 샘플](https://hogrefe.se/userfiles/files/RIAP5IR_SAMPLE.pdf)은 Cn을 별도 빈도로 제시하면서 S-CON 조건을 `CF + C > FC`로 적습니다. 다만 공개 샘플의 Cn이 0이므로, Cn이 있을 때 포함되는지까지 이 샘플만으로 판별할 수는 없습니다.
- CHESSSS 공개 설명서는 Cn 입력과 Color-Shading 출력의 존재를 설명하지만, 이번 검산에서 재현 가능한 공개 수식은 확보하지 못했습니다. 따라서 S-CON과 Color-Shading의 Cn 취급을 확정하는 근거로 사용하지 않았습니다.

따라서 “Cn을 항상 포함” 또는 “Cn을 항상 제외”라는 하나의 규칙은 둘 다 틀립니다. **어느 결과를 계산하는지**가 먼저입니다.

## 6. 같은 오류가 다시 생기지 않도록 한 조치

### 계산 코드

- 화면 표시용 `displayedColorRatioRight = CF + C + Cn`을 별도 변수로 둡니다.
- S-CON용 `sconUnmodulatedColor = CF + C`를 별도 변수로 둡니다.
- WSumC는 가중식 자체를 유지해 Cn이 들어갈 통로를 만들지 않습니다.
- Color-Shading은 `CHROMATIC_COLOR = [FC, CF, C]` 목록으로 별도 집계합니다.

### 대표 예시를 이용한 자동 검사

- Cn만 있는 최소 프로토콜에서 화면 비율만 변하고 WSumC, DEPI, S-CON은 변하지 않는지 확인합니다.
- `FC=2`, `CF=1`, `Cn=2`인 경계 프로토콜에서 화면은 `2:3`, WSumC는 `2.0`, S-CON c7은 false인지 동시에 확인합니다.
- 같은 blend에 Cn만 주입해 R과 blend 수를 고정한 차분 테스트에서 화면 우변만 1 증가하고 EB·WSumC·S-CON c2/c7·DEPI c2가 그대로인지 확인합니다.
- 공개 완성 사례에서 `FC=2`, `CF+C=4`, `Cn=1`의 화면 결과가 `2:5`인지 확인합니다.
- 2,000개 seed 고정 합성 프로토콜 오라클도 화면 비율에 Cn을 포함하도록 독립 계산합니다.

### AI 답변 확인 기준

이전 문자열 판정기는 기대 신호가 `2:4`여도 응답 안에 따로 등장한 숫자 2와 4를 순서대로 찾아 통과시킬 수 있었습니다. 예를 들어 “FC=2, CF=4이며 실제 비율은 2:5” 같은 정답도 잘못된 기대식 `2:4`를 충족할 수 있었습니다.

v2.2.2에서는 `:`, `=`, `+`, `*`, `/`, `<`, `>`가 들어간 수식형 기대 신호를 **문자 그대로** 찾아야 합니다. 분리된 숫자 토큰으로는 통과할 수 없습니다. 코딩 도우미와 해석 도우미 각각에 Cn 경계 fixture를 두고 `2:5`, `CF+C+Cn`, Cn 포함 설명을 요구합니다.

### 5개 언어 지식층

한국어, 영어, 일본어, 스페인어, 포르투갈어에서 다음 설명을 같은 의미로 맞췄습니다.

- 화면의 관례적 `FC:CF+C` 우변은 CF+C+Cn
- WSumC는 Cn 제외
- S-CON c7은 Cn 제외
- Color-Shading은 Cn 제외

작성 원문에서 생성 코퍼스를 다시 만들고, OpenAI 임베딩 6,629개를 전량 다시 생성했습니다. 언어별 상태는 ko 1,403, en 1,283, ja 1,283, es 1,405, pt 1,255이며 stale row는 0입니다. 전 언어 prune이 locale을 무시해 다른 언어와 같은 chunk ID를 가진 폐기 행을 남길 수 있던 운영 결함도 언어별 비교로 교정하고 회귀 테스트를 추가했습니다.

## 7. 다섯 언어 예시의 정확한 의미

계산 함수는 자연어 메모를 읽지 않습니다. Card, Location, DQ, Determinants, FQ, Pair, Contents, P, Z, Special Scores만 계산합니다. 따라서 같은 부호 입력이라면 언어와 관계없이 결과가 같아야 합니다.

| 언어 | 같은 Cn 기전을 설명하는 표현 | 가능한 후보 |
| --- | --- | --- |
| 한국어 | 이건 빨간색이에요. 모양은 모르겠고 그냥 빨간색입니다. | `Cn`, FQ none |
| English | It is red. I do not see a shape; it is just the color red. | `Cn`, FQ none |
| 日本語 | 赤い色です。形は分からず、ただ赤い色に見えます。 | `Cn`, FQ none |
| Espanol | Es de color rojo. No veo una forma; solo es el color rojo. | `Cn`, FQ none |
| Portugues | E vermelho. Nao vejo uma forma; e apenas a cor vermelha. | `Cn`, FQ none |

이 번역은 문화권별 규준을 검증한다는 뜻이 아닙니다. 동일한 부호 입력이 언어 메모 때문에 다른 산술 결과를 내지 않는지 확인하는 **언어 불변성 테스트**입니다. 실제 Cn 판정에는 원반응과 Inquiry가 필요합니다.

## 8. AI 플레이북, 코퍼스, 임베딩의 정정 범위

계산 코드만 고치고 AI 지식층을 그대로 두면 도우미가 과거 설명을 다시 말할 수 있습니다. 그래서 다음을 함께 동기화했습니다.

- 5개 언어 `FC:CF+C` 작성 원문
- 런타임 변수 설명 `result-variables.ts`
- 생성된 route 문서와 chunk 코퍼스
- 코딩·해석 도우미 eval fixture
- 수식형 문자열 판정기
- OpenAI 벡터 임베딩과 release snapshot

AI 지식층에는 계산 경계를 설명하는 운영형 요약만 들어갑니다. 비공개 보유 자료의 파일명, 로컬 경로, 원문, 스캔 이미지 또는 긴 인용을 공개 코퍼스로 옮기지 않습니다.

## 9. 공개 범위와 개인정보·보안 경계

공개 저장소에는 다음을 남깁니다.

- 교정된 계산 코드와 회귀 테스트
- 5개 언어 계산 설명 문서
- 계산 계보와 오류 발동 조건
- raw 대화가 없는 live eval 집계 메타데이터
- 사용한 공개 자료의 canonical 링크

공개하지 않는 항목:

- 비공개 보유 자료의 로컬 파일명, 경로, 해시, 원문 또는 스캔
- API key와 `.env*`
- Neon/Vercel 연결 정보
- raw prompt, raw model answer, chat message
- 반응 메모, 구조요약 private payload, patient data
- 사설 저장소 commit hash

공개 소스를 만드는 과정에서는 AI 검사 JSON과 JSONL에 남을 수 있는 비공개 버전 식별값도 제거합니다.

## 10. 남아 있는 한계

- 자동 테스트는 알려진 규칙과 생성 조합을 넓게 검사하지만 가능한 모든 인간 반응을 수학적으로 증명하지 않습니다.
- 계산기는 입력된 부호를 집계합니다. 잘못 부호화된 입력을 임상적으로 교정하지 않습니다.
- 공개 구현끼리 일치한다는 사실만으로 판본 규칙이 되지는 않습니다.
- S-CON criterion 7과 Color-Shading에서 Cn을 제외하는 경계는 이 앱이 채택한 판본과 구현 범위의 명시적 계약입니다. 공개 재현 자료는 일부가 침묵하거나 서로 달라, 외부 프로그램 전체가 같은 식을 사용한다고 일반화하지 않습니다.
- CHESSSS 배포 파일의 셀 수식은 이번 검산에서 공개 경로로 다시 확보하지 못했습니다. 공개 설명서에서 직접 확인할 수 없는 수식 주장은 근거에서 제외했습니다.
- 언어 불변성은 산식 동일성을 뜻하며 문화적 규준 타당성 인증이 아닙니다.
- GPT-5.5 live eval은 검색·응답 경로의 엔지니어링 검증이며 임상 인증이 아닙니다.
- v2.2.2의 UI 변경은 좌측 패널을 불투명하게 바꾼 것뿐입니다. 모바일 UI/UX 최적화는 후속 v2.2.x 범위입니다.
