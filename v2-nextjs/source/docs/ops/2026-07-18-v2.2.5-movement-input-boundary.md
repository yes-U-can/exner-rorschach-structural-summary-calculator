# v2.2.5 운동 결정인 입력 경계 검증

## 먼저 확인할 내용

이 문서는 채점표의 운동 결정인 입력에서 `M`, `FM`, `m`을 제거한 이유와 계산 영향 범위를 설명합니다.

`M`, `FM`, `m`은 구조요약에서 사람운동, 동물운동, 무생물운동의 **전체 빈도를 표시하는 합계 항목**으로 필요합니다. 그러나 개별 반응을 부호화할 때에는 운동이 능동인지, 수동인지, 능동과 수동이 함께 있는지를 반드시 구분해야 합니다.

따라서 개별 반응 입력에는 다음과 같이 완성된 부호를 사용합니다.

| 운동 종류 | 능동 | 수동 | 능동-수동 |
| --- | --- | --- | --- |
| 사람운동 | `Ma` | `Mp` | `Ma-p` |
| 동물운동 | `FMa` | `FMp` | `FMa-p` |
| 무생물운동 | `ma` | `mp` | `ma-p` |

v2.2.4까지는 채점 드롭다운에 완성된 부호와 함께 능동형·수동형 표기가 없는 `M`, `FM`, `m`도 나타났습니다. v2.2.5에서는 이 세 값을 **입력 선택지에서만 제거**했습니다. 구조요약 결과의 `M`, `FM`, `m` 합계와 이를 사용하는 EB, MQual, W:M 등의 표시는 그대로 유지합니다.

## 어떤 기록이 영향을 받을 수 있나요?

능동형·수동형을 올바르게 구분한 기록은 영향을 받지 않습니다. `Ma`, `Mp`, `Ma-p`, `FMa`, `FMp`, `FMa-p`, `ma`, `mp`, `ma-p`만 입력한 기존 프로토콜은 이 문제 때문에 다시 계산할 필요가 없습니다.

영향 가능성이 있는 것은 과거 버전에서 개별 반응의 결정인으로 능동형·수동형 표기가 없는 `M`, `FM`, `m`을 직접 선택한 기록뿐입니다. 과거 입력 빈도를 수집하지 않았기 때문에 실제 발생 횟수는 알 수 없습니다.

이 문제는 겉으로 알아채기 어려웠습니다. 능동형·수동형 표기가 없는 값을 입력해도 운동의 전체 빈도와 EB, MQual 같은 값은 올라갔지만, 능동·수동 빈도에는 들어가지 않았기 때문입니다. 그 결과 다음 값이 낮게 계산될 수 있었습니다.

- Lower Section의 `a:p`
- Lower Section의 `Ma:Mp`
- 대인관계 영역의 능동·수동 빈도
- 위 능동·수동 빈도를 사용하는 CDI의 네 번째 조건

## CDI가 달라질 수 있는 경계 예

아래 예는 오류가 발동하는 계산 경계를 가장 작게 보여주기 위한 가상 자료입니다. 두 반응뿐이므로 실제 임상 해석에 사용할 수 있는 완성 프로토콜이 아닙니다.

1. “사람이 쉬고 있습니다.”를 `Mp`, `H`로 부호화합니다.
2. “다른 사람이 누워 잠들어 있습니다.”도 `Mp`, `H`로 부호화합니다.

두 운동을 모두 `Mp`로 입력하면 수동운동은 2, 능동운동은 0입니다. 따라서 CDI의 네 번째 조건인 `수동운동 > 능동운동 + 1`은 `2 > 1`이 되어 충족됩니다. 이 경계 자료에서는 CDI가 `4, Positive`로 계산됩니다.

과거 버전에서 두 번째 `Mp`를 능동형·수동형 표기가 없는 `M`으로 잘못 입력하면 사람운동 전체 빈도는 여전히 2로 보입니다. 그러나 수동운동은 1로만 집계되어 `1 > 1`이 거짓이 됩니다. 다른 세 조건이 같은 경계 자료에서는 CDI가 `3, NO`로 달라질 수 있습니다.

이 예는 부호화 판단을 자동화하려는 것이 아닙니다. 반응기록과 질문단계(Inquiry)를 바탕으로 `a`, `p`, `a-p`를 정하는 책임은 임상가에게 있습니다. 앱의 책임은 임상가가 입력한 완성 부호를 빠뜨리지 않고 계산하는 것입니다.

## 이 입력이 생긴 경로를 어떻게 확인했나요?

초기 개발 자료와 공개 소스를 다시 대조했습니다.

### 확인된 사실

- 초기 v1 개발에 참고한 [2019년 Excel 계산 파일의 공개 배포 위치](https://blog.naver.com/jin_k84/221539279596)를 확인한 결과, 입력 목록에는 능동형·수동형이 구분된 운동 부호만 있었습니다. 능동형·수동형 표기가 없는 `M`, `FM`, `m`은 입력값이 아니라 이 운동 부호들을 합산한 결과였습니다.
- [RorScore의 Perl 구현](https://github.com/jleader/RorScore/blob/master/bin/rorschach.pl)은 능동형·수동형이 구분된 운동 부호를 읽은 뒤 `M`, `FM`, `m` 합계와 `a:p` 빈도를 따로 만들었습니다. 이 코드의 `M`, `FM`, `m`도 입력 선택지가 아니라 내부 합계 항목이었습니다.
- v1.0.0 채점 드롭다운에는 개별 반응에 넣는 부호와 구조요약의 합계 항목이 함께 들어갔고, 이후 v2까지 이어졌습니다.
- v1.0.2에서는 MQual 계산을 고치면서 능동형·수동형 표기가 없는 `M`을 `Ma`, `Mp`, `Ma-p`와 같은 개별 반응 부호처럼 추가한 기록이 남아 있습니다.

### 단정하지 않은 부분

Excel의 입력 목록과 Perl의 합계 항목이 어떤 수작업 순서로 합쳐졌는지는 남은 기록만으로 확정할 수 없습니다. 두 자료에서 서로 다른 역할을 하던 항목이 v1에서 함께 노출됐다는 것은 확인할 수 있지만, 이를 기계적으로 복사·병합했다고 단정하지 않습니다.

이번 문제는 Excel이나 RorScore의 계산을 그대로 옮긴 결과라기보다, **개별 반응에 넣는 부호와 구조요약에서 보여주는 합계 항목의 구분이 v1 구현에서 흐려진 상태로 이어진 것**으로 보는 것이 가장 타당합니다. 이 문장은 개발 기록을 바탕으로 한 해석이며, 위의 직접 확인 사실과 구분합니다.

## 외부 기준과의 대조

한 자료만 정답으로 삼지 않고, 다음 공개 자료에서 운동 반응에 능동·수동 구분이 필요하다는 점을 함께 확인했습니다.

- [CHESSSS 1 공식 설명서](https://www.rorschach-institute.org/CHESSSS%201%20HANDBOOK%20%28English%29.pdf): 결정인 입력 목록에 능동형·수동형이 구분된 운동 부호만 제시합니다.
- [RAP 결정인 안내](https://virtualpsychology.com/documentation/rap/code_help/determinants.html): 모든 운동 반응에 능동 또는 수동 부호를 지정해야 한다고 설명합니다.
- 초기 2019 Excel의 입력 목록, RorScore의 집계 방식, 현재 앱의 계산 결과를 각각 따로 대조했습니다.

이 자료들은 서로 다른 프로그램과 설명서입니다. 서로 일치하는 부분은 교차 확인에 사용했지만, 어느 하나를 Exner 원전 전체를 대신하는 단독 정답으로 취급하지 않았습니다.

## 수정한 내용

1. 채점 드롭다운은 29개의 완성된 결정인 부호만 제공합니다.
2. `M`, `FM`, `m`은 구조요약의 합계 항목과 출력 표기에 그대로 남겼습니다.
3. 과거 자동저장 자료에서 능동형·수동형 표기가 없는 값이나 등록되지 않은 결정인을 발견하면 원본을 바꾸지 않고 계산을 멈춥니다.
4. 경고에는 다시 확인해야 할 행 번호와 부호를 표시합니다. `M`을 임의로 `Ma`나 `Mp`로 바꾸지 않습니다.
5. 다섯 언어의 운동 결정인 참조 문서와 화면 내 보조 설명을 같은 기준으로 고쳤습니다.
6. 코딩 도우미가 `M`, `FM`, `m`을 완성된 개별 반응 부호로 제시하지 않고, `a`, `p`, `a-p` 정보를 확인하도록 답변 규칙을 보강했습니다.
7. 다섯 언어 문서의 `입력 코드 규칙` 절을 별도 검색 단위로 보존해, 기존 `채점/적용 조건` 절과 섞이지 않도록 생성 규칙과 회귀 검사를 보강했습니다.

## 확인 결과

- 유효 입력 목록과 계산기가 같은 29개 결정인 목록을 공유하도록 바꿨습니다.
- 능동형·수동형 표기가 없는 `M`, `FM`, `m`, 과거 Excel 표기인 `m'a`, 등록되지 않은 임의 부호가 조용히 계산되지 않는지 확인했습니다.
- 구조요약의 `M`, `FM`, `m` 전체 빈도, 단일 결정인(Single), 혼합 결정인(Blends)이 서로 다른 의미로 계속 계산되는지 확인했습니다.
- 위 CDI 경계에서 두 개의 유효한 `Mp`가 네 번째 조건을 충족하고, 한 개만 있을 때는 충족하지 않는지 반복 검사에 고정했습니다.
- 다섯 언어의 참조 문서 검색 질문 365개가 모두 관련 문서를 찾았습니다.
- OpenAI 임베딩 5,604개를 새 본문으로 다시 만들었고, 오래된 임베딩과 본문 해시 불일치는 0개였습니다.
- 실제 OpenAI 임베딩을 사용한 혼합 검색에서 넓은 질문의 Hit@1·Hit@8, 이름을 지정한 질문의 Hit@4, 관련 문서 묶음 Coverage@8이 모두 1.0이었고, 넓은 질문의 다른 작업영역 오염@1은 0이었습니다.
- 실제 GPT-5.5 단일 대화 62회와 여러 차례 대화 9회에서 운동의 능동·수동 부호에 관한 답변 기준을 확인했습니다. 이 과정에서 운동 부호와 무관한 두 검사 기준의 취약점도 발견해 각각 반복 확인으로 바로잡았습니다.
- 다섯 언어 문구와 임베딩을 최종 정리한 뒤, 운동 결정인의 입력 경계를 묻는 대표 질문을 언어별로 한 번씩 다시 호출했습니다. 5회 모두 통과했고, 중단과 기준 위반은 0건, 추정 비용은 `$0.156525`였습니다. 공개 가능한 요약 기록은 [최종 다섯 언어 실제 호출 결과](../ai-evals/live-eval-v2.2.5-movement-boundary-final-language-2026-07-18.jsonl)에서 확인할 수 있습니다.
- 릴리즈 전 독립 감사에서 참조 문서 절 제목 생성 규칙을 보완한 뒤 같은 다섯 언어 질문을 다시 호출했습니다. 5회 모두 통과했고, 중단과 기준 위반은 0건, 추정 비용은 `$0.159545`였습니다. 공개 가능한 요약 기록은 [독립 감사 후 다섯 언어 실제 호출 결과](../ai-evals/live-eval-v2.2.5-movement-boundary-post-opus-2026-07-18.jsonl)에서 확인할 수 있습니다.
- 웹앱과 같은 API 경로를 사용한 4개 확인에서도 응답 완료, 낮은 R 경계, 연령이 필요한 S-CON 질문, Form Quality 확인 질문이 모두 통과했습니다.
- 전체 자동 검사에서는 81개 테스트 파일의 447개 검사가 통과했고 7개는 실행 조건이 없어 건너뛰었습니다. 배포용 화면 222개도 모두 생성됐으며, lint와 다섯 언어 문구 감사, 비밀정보 검사, 프로덕션·개발 의존성 감사도 통과했습니다.

실제 API 호출 검사는 준비한 질문에서 답변 경계가 작동하는지 확인한 것입니다. 모든 실제 임상 질문의 정확성을 보증하지 않으며, 구조요약 계산의 정답을 AI 답변으로 판정하지 않습니다.

## 개인정보와 화면 변경

- 새로운 개인정보를 수집하지 않습니다.
- 피드백 데이터베이스 구조를 바꾸지 않습니다.
- 채점 자료는 기존과 같이 브라우저에 임시 저장되며, 서버 데이터베이스에 저장하지 않습니다.
- 새 화면이나 새 입력 항목을 만들지 않았습니다. 드롭다운에서 무효한 세 선택지를 제거하고, 과거 자동저장 자료에 무효한 값이 있을 때 기존 알림 방식으로 재확인을 요구한 것이 화면 동작의 전부입니다.
