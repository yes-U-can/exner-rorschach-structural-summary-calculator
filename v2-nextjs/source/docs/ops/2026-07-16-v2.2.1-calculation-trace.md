## 먼저 고정한 제품 경계

- API 키가 없는 세션에서 이 제품은 **연령을 받지 않는 구조요약 계산기**다.
- 계산기는 사용자가 입력한 부호를 집계하고 구조요약 값을 산출한다. 반응을 어떻게 부호화하고 결과를 어떻게 임상적으로 해석할지는 전문가인 사용자의 책임이다.
- 연령처럼 일부 해석에만 필요한 정보는 계산기 입력값으로 강제하지 않는다.
- AI 해석 도우미가 연령 제한이 있는 지표의 해석을 요청받았고 연령이 제공되지 않았을 때만, 대화 안에서 이유를 짧게 설명하고 연령을 질문한다. 값을 추정하거나 계산기 UI에 추가하도록 요구하지 않는다.

## 다섯 질문에 대한 결론

### 1. 원본 Excel이 잘못되었는가?

**일부 수식은 맞고, 일부는 잘못되었다.** Excel 전체가 무가치한 것은 아니다. WSumC, EA, es, AdjEs, 자기중심성 지수의 반사반응 3배 가중 등 많은 핵심 수식은 맞았다. 그러나 다음은 이 앱이 기준으로 삼은 Exner 구조요약 규칙과 달랐다.

1. `Lower_Section!M12`와 `M13`은 D와 AdjD를 `-5`부터 `+5`로 고정했다. Exner 본문은 `-2.5`부터 `+2.5`까지를 0으로 두고, 그 밖에서는 **추가 2.5점마다 계속** 1단계씩 변한다고 명시한다. 개발 과정에서는 CHESSSS 배포 파일의 상한 없는 계산도 비교했지만, 그 셀 수식은 최종 독립 감사에서 같은 공개 경로로 다시 확보하지 못했으므로 최종 수식 근거로 인용하지 않는다.
2. `Lower_Section!M11`은 EB의 양쪽이 0이 아니면 항상 큰 값/작은 값 비율을 표시했다. 공식 EBPer에는 `EA >= 4`, `Lambda < 1`, 그리고 EA 구간별 EB 양쪽 차이 조건이 모두 필요하다.
3. Afr와 WDA%는 분모가 0인 입력에서 Excel 오류가 된다. 이 경우 값은 0이 아니라 정의 불가다.

### 2. Excel 오류는 임상적으로 얼마나 특수한 경우에 나타나는가?

오류마다 범위가 다르다. 공개된 발생률 자료가 없으므로 임의의 퍼센트는 만들지 않고 **발동 조건**으로 설명한다.

| 오류 | 발동 조건 | 임상적 특수성 |
| --- | --- | --- |
| D/AdjD 상한 | `EA-es > +15.0` 또는 `< -15.0`; AdjD는 `EA-AdjEs`에 같은 조건 | 매우 큰 자원-부담 불균형이 필요한 극단 구간이다. 일반 기록보다 드물지만, 발생하면 값의 방향은 같아도 심한 정도가 축소된다. |
| EBPer 무조건 표시 | EB 양쪽이 양수이지만 EA, Lambda 또는 절대 차이 조건 중 하나라도 실패 | 희귀 부호가 없어도 생긴다. 보통의 유효 프로토콜에서도 가능한 조건이므로 D 상한보다 훨씬 넓다. |
| WDA% 분모 0 | W, WS, D, DS 반응이 하나도 없음 | 표준 실시에서 매우 이례적이다. 모든 반응이 Dd 계열인 경우처럼 극단적인 위치 분포에서만 생긴다. |
| Afr 분모 0 | I-VII 카드 반응이 하나도 없음 | 표준 카드 실시를 전제로 하면 사실상 비정상 입력 또는 불완전 기록에 가깝다. |
| FC:CF+C의 Cn 경계 | Cn이 한 개 이상 있음 | Cn 자체가 드물어 적용 범위는 좁다. Cn은 표시 비율 우변에만 더하고 WSumC·S-CON·Color-Shading에는 더하지 않아야 한다. |

D 오류는 한 문장 하나로 생기지 않고 프로토콜 전체의 합으로 생긴다. 예를 들어 `EA=3.5`, `es=19.0`이면 원차는 `-15.5`다. Excel은 `D=-5`로 멈추지만 연속 변환은 `D=-6`이다. `m`, `Y`, `C'`, `T`, `V`, `FM` 계열 결정인이 여러 반응에 누적되면 es가 커질 수 있다.

기전 설명용 반응 예시는 “어두운 구름이 바람에 밀려 내려오고 있어요”다. Inquiry에서 무생물의 움직임과 명암 사용이 실제로 확인된다면 `m` 계열과 `Y` 계열 후보가 될 수 있다. 단, 이 한 문장만으로 코드를 확정하거나 D=-6을 만들 수는 없다. 정확한 결정인·형태질은 반점 영역과 Inquiry를 확인한 인간 채점자가 결정한다.

EBPer 오류는 공식 Chapter 9 예제가 가장 분명하다. `M=7`, `WSumC=4`, `EA=11`, `Lambda=.55`이고 양쪽 차이가 3이므로 EBPer는 `7/4=1.75`다. v2.2.0은 비율 자체가 2.5 이상이어야 한다는 잘못된 조건 때문에 이 값을 숨겼다.

**v1과 v2.2.0에 실제 오류가 있었다.** 다만 Excel을 그대로 복사한 것만은 아니며, 버전마다 계보가 다르다.

| 항목 | 2019 Excel | v1 GAS | v2.2.0 | v2.2.1 |
| --- | --- | --- | --- | --- |

| GHR/PHR | 별도 Excel 판정 | 순서 규칙 일부 축약 | 축약 계승 | 7단계 순서 함수로 단일화 |

| WDA%/Afr 0분모 | Excel 오류 | 0으로 대체 | 0으로 대체 | 정의 불가 `-`, 지표 조건에 0으로 투입하지 않음 |

| all-F Lambda | 별도 문구 | 0으로 대체 | v2.2.0 전에 무한대로 교정됨 | 무한대 회귀 테스트 유지 |

### 4. 앱 오류는 임상적으로 얼마나 특수한 경우에 나타나는가?

- **D/AdjD 상한**은 원차가 +/-15를 넘어야 하므로 좁고 극단적인 경우다. 그러나 그런 기록에서 심한 정도를 한 단계 이상 축소할 수 있어 계산기는 정확히 처리해야 한다.
- **EBPer**는 특정 희귀 부호가 필요하지 않다. 공식 예제처럼 `EA>10`, 양쪽 차이는 충분하지만 비율이 2.5 미만인 평범한 구조에서도 v2.2.0이 값을 누락했다. 이번 결함 중 실제 적용 범위가 가장 넓다.
- **GHR/PHR**은 인간표상 반응에 좋은 FQ와 `ALOG`, `CONTAM` 또는 Level 2 인지 특수점수가 함께 있을 때 나타난다. 흔한 조합은 아니지만 임상적으로 가능한 명시 규칙이다.
- **Cn 표시 누락**은 Cn이 있어야 하므로 비교적 좁다. 영향은 `FC:CF+C` 표시 우변에 한정된다. Cn은 WSumC, S-CON의 `CF+C>FC`, Color-Shading Blend에는 넣지 않는다.
- **WDA%/Afr 0분모**는 표준 실시에서는 매우 이례적인 입력이다. 핵심은 이를 정상적인 0점으로 오인하지 않는 것이다.

GHR/PHR 기전 예시: “두 사람이 서로 밀고 있어요. 둘이 떨어져 있으니 같은 사람인 게 분명해요.”라는 응답에서 영역·형태·운동이 확인되고 마지막 설명이 부적절한 논리로 판정된다면 `Do Ma H ALOG` 같은 후보가 될 수 있다. 좋은 FQ의 인간표상이어도 ALOG가 있으면 7단계 중 Step 2에서 PHR이다. 이 예시는 계산 경로를 설명하기 위한 것이며 실제 최종 부호화는 Inquiry와 Form Quality 검토가 필요하다.

Cn 기전 예시: “이건 빨간색이에요. 모양은 모르겠고 그냥 빨간색입니다.”가 순수한 색 이름 반응으로 확인되면 `Cn`, FQ none 후보다. Cn 한 개는 화면의 `FC:(CF+C+Cn)` 우변에 1을 더하지만 WSumC에는 더하지 않는다.

### 5. 맞고 틀림을 평가한 기준은 무엇인가?

1. Exner 5판 Chapter 9의 직접 계산 문장과 표를 코드 수준의 조건식으로 옮겼다.
2. Chapter 7의 GHR/PHR 7단계는 **먼저 맞은 단계에서 종료**되는 순서를 그대로 독립 함수와 단계별 테스트로 만들었다.
3. Exner Chapter 9의 17반응 완성 예제를 처음부터 끝까지 재계산해 상단, 하단, 여섯 특수지표를 대조했다.
4. CHESSSS 공개 설명서와 배포 페이지를 읽기 전용으로 확인했다. 설명서는 EBPer를 고려하지 않아야 하는 사례에서도 값을 표시할 수 있다는 한계를 명시한다. 앞선 감사에서 기록한 배포 파일의 셀 수식은 최종 독립 감사에서 같은 공개 경로로 다시 확보하지 못했으므로, v2.2.2의 최종 수식 근거에서는 제외했다.
5. 2019 Excel의 실제 셀 수식, v1 GAS, v2.2.0 TypeScript를 나란히 놓고 오류가 어디서 생기고 어디까지 계승되었는지 추적했다.
6. 공개 점수열 3건, Exner 완성 예제 1건, 원본 Excel 내장 사례 1건을 회귀 fixture로 사용했다.
7. 별도 계산 오라클로 2,000개의 재현 가능한 합성 프로토콜을 검사했고, PTI·DEPI·CDI·S-CON·HVI·OBS의 42개 조건이 각각 true와 false를 모두 지나도록 확인했다.

## 다섯 언어 검증의 정확한 의미

구조요약 계산 함수는 반응 메모의 자연어를 읽지 않고 Card, Location, DQ, Determinants, FQ, Pair, Contents, P, Z, Special Scores만 계산한다. 따라서 한국어·영어·일본어·스페인어·포르투갈어에서 수식이 달라져서는 안 된다.

다섯 개의 점수열 fixture 각각에 다섯 언어 메모를 넣어 총 25회 계산하고, 메모를 제외한 모든 계산 결과가 완전히 동일한지 검사한다. 이는 **언어 불변성 테스트**이지, 다섯 문화권의 규준 타당성을 증명하는 테스트가 아니다. 지역별 사례 수를 꾸며 “문화적 검증”으로 부르지 않는다.

Cn 예시의 다섯 언어 표현은 다음과 같다. 어느 언어든 부호 입력이 같으면 산술 결과도 같아야 하며, Cn 한 개를 추가하면 표시 우변만 1 증가해야 한다.

| 언어 | 기전 설명용 응답 | 가능한 후보 |
| --- | --- | --- |
| 한국어 | 이건 빨간색이에요. 모양은 모르겠고 그냥 빨간색입니다. | `Cn`, FQ none |
| English | It is red. I do not see a shape; it is just the color red. | `Cn`, FQ none |
| 日本語 | 赤い色です。形は分からず、ただ赤い色に見えます。 | `Cn`, FQ none |
| Espanol | Es de color rojo. No veo una forma; solo es el color rojo. | `Cn`, FQ none |
| Portugues | E vermelho. Nao vejo uma forma; e apenas a cor vermelha. | `Cn`, FQ none |

번역문은 같은 계산 기전을 설명하기 위한 예시다. 실제 Cn 판정에는 원반응과 Inquiry가 필요하다.

## GPT-5.5와 RAG 실전 검증

계산식 테스트와 AI 응답 테스트는 서로 다른 책임을 확인한다. 계산식의 정답 여부는 결정적 TypeScript 테스트와 독립 오라클로 판정하고, GPT-5.5 호출은 교정된 참조 문서를 실제 도우미가 검색하고 경계를 지키며 끝까지 답하는지를 확인한다. LLM 응답을 계산식의 정답지로 사용하지 않았다.

1. 실제 웹앱 `/api/chat` 경로에서 영어 저반응수 해석, 한국어 장문 해석, 한국어 S-CON 연령 미제공, 한국어 모호한 FQ 코딩 질문을 호출했다. 네 건 모두 HTTP 200, `gpt-5.5`, 정상 종료였고 장문 응답도 4,031자에서 중단되지 않았다.
2. 한국어·영어·일본어·스페인어·포르투갈어에서 해석과 코딩 workflow를 섞은 다섯 건을 호출했다. 첫 실행은 새 코퍼스와 기존 벡터 스냅샷의 지문 불일치 때문에 안전한 lexical fallback으로 동작했고, release gate가 이를 실패로 잡았다.
3. 6,632개 OpenAI 임베딩을 현재 코퍼스로 다시 생성했다. 언어별 청크는 ko 1,405, en 1,283, ja 1,283, es 1,406, pt 1,255이며 stale row는 모두 0이다.
4. 같은 다섯 건을 다시 실행했다. 전부 hybrid retrieval, vector hit 2~7개, issue 0, incomplete 0으로 통과했다. 최종 다섯 건의 추정 API 비용은 `$0.136955`였다.
저장한 JSONL은 fixture ID, 모델, workflow, token 사용량, 검색 mode, route, issue 유형 같은 집계 메타데이터만 포함한다. API key, raw prompt, raw model answer, 구조요약 원문은 저장하지 않는다.
