# [2026-07-19] v2.2.5 버그 패치

## 주요 수정사항

### 개요

채점표의 **[Determinants]** 드롭다운에서 능동형·수동형 구분이 빠진 `M`, `FM`, `m`을 선택할 수 있었던 문제를 수정했습니다.

이 세 기호는 구조요약에서 사람운동, 동물운동, 무생물운동의 전체 빈도를 표시하는 **합계 항목**으로 필요합니다. 그러나 개별 반응의 운동 결정인은 능동형 `a`, 수동형 `p`, 능동-수동형 `a-p` 중 하나로 구분해 기록해야 합니다.

따라서 v2.2.5에서는 `M`, `FM`, `m`을 **입력 선택지에서만 제거**했습니다. 구조요약 결과의 `M`, `FM`, `m` 합계와 EB, MQual, W:M 등의 계산은 그대로 유지됩니다.

올바르게 `Ma`, `Mp`, `Ma-p`, `FMa`, `FMp`, `FMa-p`, `ma`, `mp`, `ma-p`를 사용한 기존 프로토콜은 이 문제의 영향을 받지 않으며 다시 계산할 필요가 없습니다.

과거 버전에서 개별 반응에 능동형·수동형 표기가 없는 `M`, `FM`, `m`을 직접 선택했다면 다시 확인해야 합니다. 과거 자동저장 자료에서 이런 값을 발견하면 앱은 원래 입력을 보존한 채 계산을 멈추고, 확인이 필요한 행과 부호를 알려줍니다. 능동형·수동형 채점은 반응기록과 질문단계(Inquiry)를 확인해야 정할 수 있으므로 앱이 임의로 바꾸지 않습니다.

### 왜 이 문제가 중요했나요?

능동형·수동형 표기가 없는 값을 입력해도 운동의 전체 빈도, EB, MQual 등은 정상처럼 올라갔습니다. 반면 능동·수동 정보는 빠져 `a:p`, `Ma:Mp`, 대인관계 영역의 능동·수동 빈도가 실제보다 낮게 계산될 수 있었습니다.

특히 CDI의 네 번째 조건은 `수동운동 > 능동운동 + 1`을 확인합니다. 판정 경계에서는 능동형·수동형 구분 하나가 빠진 것만으로 CDI의 양성 여부가 달라질 수 있습니다.

예를 들어 “사람이 쉬고 있습니다”와 “다른 사람이 누워 잠들어 있습니다”라는 두 반응을 모두 `Mp H`로 부호화한 가상 자료를 사용했습니다. 두 운동이 모두 `Mp`이면 수동 2, 능동 0이므로 네 번째 조건이 충족되고, 화면에는 CDI가 `4, Positive`로 표시됩니다.

과거 버전에서 두 번째 `Mp`를 능동형·수동형 표기가 없는 `M`으로 입력하면 사람운동 전체 빈도는 여전히 2로 보이지만, 수동 빈도는 1로만 집계됩니다. 판정 경계에 놓인 같은 자료에서는 네 번째 조건이 충족되지 않아 CDI가 `3, NO`로 달라질 수 있었습니다.

이 예는 계산 경계를 보여주기 위해 두 개의 반응만으로 만든 가상 자료이며, 실제 임상 해석에 사용할 수 있는 완성 프로토콜이 아닙니다. 운동의 능동·수동 부호는 임상가가 반응기록과 질문단계(Inquiry)를 확인한 뒤 결정합니다.

### 어디에서 시작된 문제였나요?

초기 개발 자료를 다시 대조했습니다.

- v1 개발에 참고한 2019년 Excel 계산 파일의 입력 목록에는 능동형·수동형이 구분된 운동 부호만 있었고, `M`, `FM`, `m`은 이 부호들을 합산한 결과로 사용했습니다.
- RorScore의 Perl 구현도 능동형·수동형이 구분된 운동 부호를 읽은 뒤 `M`, `FM`, `m`의 합계와 능동·수동 빈도를 따로 계산했습니다.
- v1.0.0 채점 드롭다운에는 개별 반응에 넣는 부호와 구조요약의 합계 항목이 함께 들어갔고, 이 상태가 v2까지 이어졌습니다.
- v1.0.2의 MQual 수정 기록에서는 능동형·수동형 표기가 없는 `M`을 `Ma`, `Mp`, `Ma-p`와 같은 개별 반응 부호처럼 다룬 흔적을 확인했습니다.

Excel의 입력 목록과 Perl의 합계 항목이 정확히 어떤 수작업 순서로 합쳐졌는지는 남은 기록만으로 단정할 수 없습니다. 확인할 수 있는 범위에서는 **개별 반응에 넣는 부호와 구조요약에서 보여주는 합계 항목의 구분이 v1 구현에서 흐려진 것**이 원인에 가장 가까운 설명입니다.

Excel과 Perl은 각각의 문맥에서 개별 반응의 운동 부호와 운동 종류별 합계를 구분하고 있었습니다. 확인된 문제는 참고 자료의 계산 규칙이 아니라, 서로 역할이 다른 항목을 앱의 한 입력 목록에서 함께 다룬 구현에 있었습니다. 개발 과정과 검토 근거의 한계는 [운동 결정인 입력 경계 검증 문서](../../source/docs/ops/2026-07-18-v2.2.5-movement-input-boundary.md)에 자세히 적었습니다.

## 함께 수정한 내용

- 다섯 언어의 `M`, `FM`, `m` 참조 문서에 구조요약의 합계 항목과 개별 반응에 넣는 부호의 차이를 명시했습니다.
- 화면 안의 보조 설명과 참조 문서를 같은 기준으로 맞췄습니다.
- 코딩 도우미가 `M`, `FM`, `m`을 개별 반응에 그대로 넣을 수 있는 부호로 제시하지 않고, `a`, `p`, `a-p` 정보를 확인하도록 답변 규칙을 보강했습니다.
- 일본어 검색 질문 4곳과 한국어 문서 관리 기록에서 깨진 글자를 복원했습니다.
- 참조 문서를 바꾼 뒤 다섯 언어 검색 자료와 OpenAI 임베딩을 모두 다시 만들었습니다.
- `입력 코드 규칙`과 `채점/적용 조건`이 서로 다른 절로 유지되도록 문서 생성 규칙과 회귀 검사를 보강했습니다.
- 넓은 구조요약 질문에서 응답 수와 자료의 한계를 먼저 설명하도록 해석 도우미의 답변 순서를 보강했습니다.

## 테스트 및 검증

입력과 결과가 서로 다른 역할을 지키는지 먼저 확인했습니다.

- 채점 드롭다운에는 완성된 결정인 부호 29개만 나타납니다.
- `M`, `FM`, `m`은 구조요약 결과의 운동 종류별 합계 항목으로 그대로 남습니다.
- 과거 자동저장 자료의 능동형·수동형 표기가 없는 값, 과거 Excel식 `m'a`, 등록되지 않은 임의 부호는 원본을 바꾸지 않고 계산을 차단합니다.
- 구조요약의 운동 전체 빈도, 단일 결정인(Single), 혼합 결정인(Blends)이 서로 다른 의미로 계속 집계됩니다.
- CDI 경계 자료에서 유효한 `Mp` 두 개가 네 번째 조건을 충족하고, 하나만 있을 때는 충족하지 않는지 반복 검사로 고정했습니다.

다섯 언어의 참조 문서와 선택형 AI 도우미도 함께 확인했습니다.

- 참조 문서 검색 질문 365개가 모두 관련 문서를 찾았습니다.
- OpenAI 임베딩 5,604개를 새 본문으로 다시 만들었고, 오래된 임베딩과 본문 해시 불일치는 0개였습니다.
- 실제 OpenAI 임베딩을 사용한 혼합 검색에서 넓은 질문과 이름을 지정한 질문의 상위 문서 적중률, 관련 문서 묶음의 포함률이 모두 100%였고, 넓은 질문에 다른 작업영역 문서가 첫 결과로 섞인 경우는 없었습니다.
- 실제 GPT-5.5 단일 대화 62회에서는 운동의 능동·수동 부호를 묻는 질문이 모두 의도한 경계를 지켰습니다. 운동 부호와 무관한 장문 해석 답변 1회에서 응답 수와 자료의 한계를 먼저 설명하지 않은 문제가 발견되어 답변 순서를 보완했고, 같은 조건을 3회 연속 다시 확인했습니다.
- 여러 차례 이어 묻는 대화 9회에서도 운동의 능동·수동 부호를 다룬 대화는 통과했습니다. 별도의 영문 검사 1회는 뜻이 같은 표현을 검사기가 알아보지 못해 실패로 기록됐으며, 허용 표현을 바로잡은 뒤 관련 질문을 2회 연속 다시 확인했습니다.
- 다섯 언어 문서의 표현과 임베딩을 최종 정리한 뒤, 운동 결정인의 입력 경계를 묻는 대표 질문을 언어별로 한 번씩 다시 호출했습니다. 한국어, 영어, 일본어, 스페인어, 포르투갈어 5회 모두 문제없이 통과했습니다.
- 문서 절 제목 생성 규칙을 보완한 뒤 같은 다섯 언어 질문을 다시 호출했습니다. 5회 모두 문제없이 통과했으며, 중단과 기준 위반은 없었습니다.
- 웹앱과 같은 API 경로를 사용한 4개 확인도 모두 통과했습니다.
- 전체 자동 검사에서는 81개 테스트 파일의 447개 검사가 통과했고 7개는 실행 조건이 없어 건너뛰었습니다. 배포용 화면 222개도 모두 생성됐으며, 코드 정적 검사와 다섯 언어 문구 감사, 비밀정보 검사, 프로덕션·개발 의존성 감사도 통과했습니다.

실제 GPT-5.5 호출은 준비한 질문에서 코딩·해석 도우미의 답변 경계가 작동하는지 확인한 것입니다. 모든 실제 질문의 임상적 정확성을 보증하지 않으며, 구조요약 계산의 정답을 AI 답변으로 판정하지도 않았습니다.

구현과 반복 검사는 OpenAI Codex를 사용했고, 공개 전 문서·계산 근거 검토에는 Claude Opus 4.8을 사용했습니다. 도구 사이의 의견 일치를 정답으로 삼지 않았으며, 공개 전문 자료, 이 앱이 채택한 CS 범위, 재현 가능한 계산 결과와 사람의 검토를 함께 기준으로 삼았습니다.

## UI/UX, 개인정보와 데이터베이스

- 새 화면이나 새 입력 항목을 추가하지 않았습니다.
- [Determinants] 드롭다운에서 무효한 세 선택지를 제거했습니다.
- 과거 자동저장 자료에 무효한 결정인이 있으면 기존 알림 방식으로 행과 부호를 안내하고 계산을 멈춥니다.
- 새로운 개인정보를 수집하지 않습니다.
- 채점 자료를 서버 데이터베이스에 저장하지 않는 기존 원칙을 유지합니다.
- 피드백 데이터베이스 구조와 요청 제한은 변경하지 않았습니다.

## 공개 범위와 보안 경계

공개 소스에는 입력 경계 검사, 계산 회귀 검사, 다섯 언어 참조 문서, 검색 자료 스냅샷과 원문을 제외한 GPT-5.5 검사 요약을 포함합니다.

초기 개발 자료의 원본 파일, 유료 또는 비공개 문헌, 로컬 경로, API 키, 환경변수, 실제 답변 원문과 비공개 검토 기록은 공개하지 않습니다. 공개 문서에서는 재현 가능한 공개 배포 위치와 공식 공개 저장소만 연결합니다.
