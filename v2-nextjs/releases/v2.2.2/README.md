# [2026-07-16] v2.2.2 Cn 경계 재검산 핫픽스

## 주요 수정사항

### 개요

v2.2.2는 앱이 표시하는 **Upper Section, Lower Section, Special Indices**의 계산식을 다시 확인한 핫픽스입니다.

먼저 중요한 점이 하나 있습니다. v2.2.1의 `FC:CF+C` 오른쪽 값은 이미 Cn을 포함해 올바르게 계산됐습니다. **완성된 프로토콜은 이 이유만으로 다시 계산할 필요가 없습니다.** 이번 패치에서는 Cn이 들어가는 화면 비율과 Cn이 들어가지 않는 WSumC, S-CON 7번 기준, Color-Shading 계산을 서로 분리했습니다. 형태질(FQ)을 아직 입력하지 않은 미완성 행이 GHR 또는 PHR로 임시 분류되던 문제도 함께 바로잡았습니다.

검산 범위는 이 앱이 실제로 보여주는 Exner Comprehensive System 구조요약입니다. 다른 로샤 체계, 연구용 확장 지표, 임상 진단 자동화는 포함하지 않았습니다.

### 세부사항

이번 패치의 핵심은 비슷해 보이는 색채 합계를 하나의 값으로 돌려 쓰지 않는 것입니다.

| 사용 위치 | v2.2.2에서 고정한 계산 |
| --- | --- |
| 화면의 관례적 라벨 `FC:CF+C` | `FC : (CF + C + Cn)` |
| WSumC | `0.5*FC + 1.0*CF + 1.5*C` |
| S-CON criterion 7 | `CF + C > FC` |
| Color-Shading blend의 chromatic color | `FC`, `CF`, `C`만 포함 |

Cn은 첫 번째 화면 비율에는 들어가지만 나머지 세 계산에는 들어가지 않습니다. 비슷해 보이는 색채 합계를 하나로 묶어 쓰지 않고, 각 계산이 필요한 부호만 따로 합산하도록 정리했습니다. 계산 코드뿐 아니라 자동 검사와 AI 도우미의 설명도 같은 기준으로 맞췄습니다.

## 이번 릴리즈에서 바뀐 것과 바뀌지 않은 것

### 바뀐 것

- 형태질(FQ)이 비어 있는 미완성 행은 더 이상 GHR 또는 PHR로 잠정 분류하지 않습니다. FQ를 실제로 입력한 뒤에만 7단계 판정 순서를 적용합니다.
- 화면의 Cn 포함 값과 S-CON의 Cn 제외 값을 서로 다른 계산으로 분리해, 한쪽의 변경이 다른 쪽에 섞이지 않도록 했습니다.
- 5개 언어의 `FC:CF+C`와 Cn 참조 문서, AI가 검색하는 자료, 실제 답변의 기대 기준을 같은 내용으로 맞췄습니다.
- 코딩 도우미와 해석 도우미에 Cn 경계를 직접 질문하고, `2:5` 같은 완성된 결과를 정확히 설명하는지 실제 GPT-5.5 호출로 확인했습니다.
- 공개하는 AI 검사 결과에 비공개 개발 식별값이 섞이지 않도록 공개 복사 과정을 보강했습니다.
- 좌측 패널의 반투명 효과를 제거하고 완전 불투명 배경으로 고정했습니다.

### 바뀌지 않은 것

- v2.2.1 배포 코드가 이미 보여주던 Cn 포함 화면값은 유지했습니다.
- 계산 화면에 연령 입력 UI를 추가하지 않았습니다.
- API 키 없이 사용하는 계산기 본체의 개인정보 최소수집 원칙을 바꾸지 않았습니다.
- AI가 부호화나 임상 해석을 자동 확정하도록 범위를 넓히지 않았습니다.
- 모바일 전용 UI/UX 최적화는 후속 v2.2.x 범위로 남겼습니다.

## 계산기를 만들 때 참고한 자료와 이번 검산에서의 역할

초기 v1 Google Apps Script 계산기를 만들 때 두 공개 자료가 중요한 학습 참고점이었습니다.

1. 공개 배포된 2019 Excel scoring workbook의 셀 수식
2. Jeremy Leader의 [RorScore](https://github.com/jleader/RorScore)와 주요 원본 파일 [`bin/rorschach.pl`](https://github.com/jleader/RorScore/blob/master/bin/rorschach.pl)

2019 workbook의 공개 배포 위치는 [Naver Blog 게시물](https://blog.naver.com/jin_k84/221539279596)이며, workbook 안에서 확인되는 제작 표기는 `[Scoring Program] _by. Ju-Ri`입니다. 이 표기만으로 실명을 추정하지 않고 원본 파일도 재배포하지 않습니다.

두 자료는 초기 계산기가 어떻게 만들어졌는지를 설명하고 구현 차이를 찾는 데 사용했습니다. 어느 하나만을 정답으로 삼지 않고, 이 앱이 채택한 규칙, 완성된 구조요약 예시, 서로 다른 프로그램의 출력, Excel·v1·v2의 실제 계산을 서로 비교했습니다.

| 자료 | 이번 검산에서 확인한 내용 | 단독 정답으로 쓰지 않은 이유 |
| --- | --- | --- |

| 2019 Excel workbook | 초기 v1 계보와 실제 셀 수식 비교 | 일부 수식은 맞고 일부는 앱 기준 규칙과 다름 |
| GPT-5.5 | 검색·설명 경로 검증 | 계산 규칙의 정답지가 아님 |

## 다섯 질문에 대한 답

### 1. 원본 Excel이 잘못되었는가?

**일부는 맞고 일부는 달랐습니다.**

맞았던 대표 항목:

- `WSumC = 0.5*FC + CF + 1.5*C`
- EA, es, AdjEs의 기본 합산
- 자기중심성 지수에서 반사반응을 3배 가중하는 구조
- 화면의 관례적 `FC:CF+C` 오른쪽 값에 Cn을 포함하는 셀 수식

앱이 선택한 규칙과 달랐던 대표 항목:

- D/AdjD를 `-5`에서 `+5`까지만 반환
- EB 양쪽이 양수이면 정해진 표시 조건을 충족하지 않아도 EBPer를 표시
- WDA%와 Afr의 분모가 0일 때 Excel 오류 발생
- S-CON과 Color-Shading에서 Cn 포함 색채 합계를 재사용

<details>
<summary><strong>Excel 셀 수식을 직접 확인하려는 분을 위한 기술 기록</strong></summary>

같은 workbook에서 직접 확인할 수 있는 셀 위치는 다음과 같습니다.

- `Upper_Section!AF8:AF11`은 `FC`, `CF`, `C`, `Cn`이고 실제 빈도는 `AH8:AH11`에 있습니다.
- 화면 비율은 `Lower_Section!Q9 = AH8`, `S9 = SUM(AH9, AH10, AH11)`이므로 오른쪽 값은 `CF+C+Cn`입니다.
- WSumC는 `Lower_Section!S11 = SUM(AH8*0.5, AH9, AH10*1.5)`이므로 Cn을 제외합니다.
- S-CON criterion 7은 `Special_Indices!D15`가 화면용 `S9 > Q9`를 그대로 비교해 Cn을 포함합니다.
- Color-Shading은 `Scoring!AF5`의 Color 목록에 Cn이 들어 있고, `AG5`의 shading 표시와 결합한 `AH5`를 `AH4`에서 집계한 뒤 `Special_Indices!D10`이 사용합니다.

</details>

따라서 “Excel 전체가 틀렸다”도 아니고 “Excel과 같으니 앱도 맞다”도 아닙니다. 셀마다 목적과 규칙을 다시 분리해야 했습니다.

### 2. Excel의 차이는 임상적으로 얼마나 특수한 경우에 나타나는가?

공개 발생률 자료가 없으므로 임의의 퍼센트를 만들지 않고 **정확한 발동 조건**으로 설명합니다.

| 항목 | 발동 조건 | 범위 판단 |
| --- | --- | --- |
| D/AdjD 상한 | `EA-es` 또는 `EA-AdjEs`가 `+15.0`보다 크거나 `-15.0`보다 작음 | 큰 자원-부담 불균형이 필요한 극단 구간 |
| EBPer 과잉 표시 | EB 양쪽이 양수지만 EA, Lambda 또는 EA 구간별 절대 차이 조건을 충족하지 않음 | 희귀 부호 없이도 가능한 비교적 넓은 범위 |
| WDA% 0분모 | W, WS, D, DS 반응이 전혀 없음 | 표준 실시에서는 매우 이례적 |
| Afr 0분모 | I-VII 카드 반응이 전혀 없음 | 표준 카드 실시라면 불완전 입력에 가까움 |
| S-CON의 Cn 오염 | Cn이 있고 `CF+C`와 FC가 criterion 경계에 가까움 | Cn 자체가 드물지만 1점 경계를 바꿀 수 있음 |
| Color-Shading의 Cn 오염 | Cn과 shading 결정인이 같은 blend 문맥에 걸림 | 매우 특수한 조합 |

D/AdjD 예시: 프로토콜 전체 합이 `EA=3.5`, `es=19.0`이면 원차는 `-15.5`입니다. 상한형 수식은 `D=-5`에서 멈추지만, 연속되는 2.5 간격 변환은 `D=-6`입니다. “어두운 구름이 바람에 밀려 내려오고 있어요” 같은 가상 응답에서 Inquiry로 무생물 운동과 diffuse shading이 확인되면 `m`, `Y` 계열 후보가 될 수 있습니다. 다만 문장 하나로 부호나 D 값을 확정하지 않으며, 여러 반응의 누적 합이 발동 조건을 만듭니다.

EBPer 예시: `M=7`, `WSumC=4`, `EA=11`, `Lambda=.55`이면 양쪽 차이가 3이므로 이 EA 구간의 표시 조건을 충족하고 EBPer는 `1.75`입니다. 단순히 비율이 2.5 이상인지 보는 방식은 이 값을 잘못 숨깁니다.

**v1과 v2.2.0에는 실제 결함이 있었습니다.** 다만 모든 결함이 Excel에서 그대로 복사된 것은 아닙니다.

| EBPer | 단순 비율 | 단순 비율 계승 | 다른 잘못된 표시 조건 | 정해진 표시 조건 | 조건 재확인 |

| 화면 `FC:CF+C` | Cn 포함 | Cn 누락 | Cn 누락 | Cn 포함 | Cn 포함 유지·변수 분리 |

v2.2.1에서 실제 배포된 계산기는 Cn 화면값을 이미 올바르게 포함했습니다. v2.2.2에서는 화면 비율, WSumC, S-CON, Color-Shading을 각각 별도의 예로 다시 계산해 서로의 색채 합계를 잘못 가져다 쓰지 않는지 확인했습니다.

### 4. 앱 결함은 임상적으로 얼마나 특수한 경우에 나타났는가?

- **D/AdjD 상한:** `+/-15`를 벗어나는 좁고 극단적인 구간에서 심한 정도를 한 단계 이상 축소할 수 있었습니다.
- **EBPer:** 이번 계보 결함 중 적용 범위가 가장 넓었습니다. 희귀 부호가 없어도 EA, Lambda, 양쪽 절대 차이의 조합에 따라 표시 여부가 달라질 수 있었습니다.
- **GHR/PHR:** 좋은 FQ의 인간표상에 `ALOG`, `CONTAM` 또는 Level 2 인지 특수점수가 함께 있는 것처럼 흔하지 않지만 가능한 조합에서 우선순위가 달라질 수 있었습니다.
- **v1·v2.2.0의 Cn 화면 누락:** Cn이 한 개 이상인 비교적 좁은 범위에서 Lower Section 비율 오른쪽 값이 작게 표시됐습니다.
- **WDA%/Afr 0분모:** 표준 실시에서는 매우 이례적이지만, 정의할 수 없는 값을 정상적인 0으로 보이지 않는 것이 중요했습니다.

GHR/PHR 기전 예시: “두 사람이 서로 밀고 있어요. 둘이 떨어져 있으니 같은 사람인 게 분명해요.”라는 가상 응답에서 영역·형태·운동이 확인되고 마지막 설명이 ALOG로 판정된다면 `Do Ma H ALOG` 같은 후보가 될 수 있습니다. 좋은 FQ의 인간표상이어도 ALOG가 있으면 7단계 중 앞선 PHR 조건이 적용됩니다. 실제 부호화는 반응기록과 질문단계(Inquiry)를 확인한 인간 채점자가 수행합니다.

Cn 기전 예시: “이건 빨간색이에요. 모양은 모르겠고 그냥 빨간색입니다.”가 순수한 색 이름 반응으로 확인되면 `Cn`, FQ none 후보입니다. `FC=2`, `CF=4`, `C=0`, `Cn=1`이라면:

- 화면 `FC:CF+C` 결과는 `2:5`
- WSumC는 `0.5*2 + 4 + 1.5*0 = 5.0`
- S-CON c7은 `4 > 2`를 판정하며 화면 우변 5를 재사용하지 않음
- Color-Shading의 chromatic color 범위에도 Cn을 넣지 않음

5. 경계값 예시, 공개된 고정 예시, 같은 조건으로 반복 가능한 가상 프로토콜 2,000개, 계산 갈림길별 대조
6. 다섯 언어 참조 문서에서 AI가 실제로 검색하고 답변하기까지의 전체 과정 확인

공개 프로그램끼리 결과가 같다는 이유만으로 규칙을 확정하지 않았습니다. 특히 S-CON 7번과 Color-Shading에서 Cn을 다루는 방식은 공개 자료의 설명이 부족하거나 서로 달랐으므로, 모든 프로그램에 통용되는 유일한 공식이라고 주장하지 않습니다. 이 문서는 현재 앱이 채택한 판본과 구조요약 범위의 계산 기준을 밝히고, 같은 오류가 다시 생기지 않는지 반복 검사합니다. GPT-5.5의 답변도 계산 정답으로 사용하지 않았습니다.

## 다섯 언어 예시와 검증 범위

계산 함수는 자연어 메모가 아니라 입력된 부호를 계산합니다. 다음 문장들은 문화권별 규준을 주장하기 위한 사례가 아니라, 같은 Cn 개념을 다섯 언어에서 설명하기 위한 **공통 가상 예시**입니다.

| 언어 | 가상 응답 | 가능한 후보 |
| --- | --- | --- |
| 한국어 | 이건 빨간색이에요. 모양은 모르겠고 그냥 빨간색입니다. | `Cn`, FQ none |
| English | It is red. I do not see a shape; it is just the color red. | `Cn`, FQ none |
| 日本語 | 赤い色です。形は分からず、ただ赤い色に見えます。 | `Cn`, FQ none |
| Español | Es de color rojo. No veo una forma; solo es el color rojo. | `Cn`, FQ none |
| Português | É vermelho. Não vejo uma forma; é apenas a cor vermelha. | `Cn`, FQ none |

실제 Cn 판정에는 반응기록과 질문단계(Inquiry)가 필요합니다. 번역문만으로 부호를 자동 확정하지 않습니다.

## AI 도우미의 설명도 같은 계산 기준으로 맞췄습니다

API 키 없이 사용하는 계산기는 AI와 무관하게 같은 계산식을 사용합니다. 다만 선택 기능인 AI 도우미가 과거 설명을 반복하지 않도록, 다섯 언어의 참조 문서와 AI가 검색하는 자료도 함께 갱신했습니다.

확인은 **참조 문서 작성 → 검색 자료 갱신 → 관련 문서 선택 → GPT-5.5 답변 → 기대한 계산식과 한계가 포함됐는지 검사**하는 순서로 진행했습니다. 마지막 단계에서는 실제 GPT-5.5 API를 호출해 코딩·해석 답변을 직접 확인했습니다.

5개 언어 문서와 6,629개의 검색 자료를 모두 새로 만들고, 언어별 옛 자료가 남지 않도록 갱신 절차도 고쳤습니다. 마지막 검사에서는 코딩 도우미와 해석 도우미가 Cn의 서로 다른 계산 범위를 구분해 설명하는지 확인했습니다.

- 영어 코딩·해석 Cn 반복: 6/6 통과, 실행 실패 0, 발견된 문제 0
- Cn 네 계산 경계의 5개 언어 재확인: 한국어·영어·일본어·스페인어·포르투갈어 각 2회, **10/10 통과**, 중단 0, 발견된 문제 0

- 기존 5개 언어 대표 경로: 5/5 통과, 실행 실패 0, 발견된 문제 0

## 공개 범위와 개인정보 경계

공개 저장소에는 교정된 계산 코드와 자동 검사, 5개 언어 계산 설명, 원문을 제외한 AI 검사 요약, 확인에 사용한 공개 링크를 남깁니다. 다음은 공개하지 않습니다.

공개 소스 복사 과정에서도 AI 검사 파일에 남을 수 있는 내부 버전 식별값을 자동으로 제거합니다.

## 판단 기준과 남아 있는 한계

계산식은 이 앱이 채택한 명시 규칙, 완성된 구조요약 예시, 실제 프로그램 출력, 반복 가능한 검사 결과를 서로 대조해 판단했습니다. 어느 한 프로그램이나 AI 답변도 단독 정답으로 사용하지 않았습니다.

자동 테스트는 알려진 규칙과 생성 조합을 넓게 검사하지만 가능한 모든 인간 반응을 수학적으로 증명하지는 않습니다. 계산기는 입력된 부호를 집계하며 잘못 부호화된 입력을 임상적으로 교정하지 않습니다. CHESSSS 배포 파일의 셀 수식은 이번 검산에서 공개 경로로 다시 확보하지 못했으므로, 공개 설명서에서 직접 확인할 수 없는 수식 주장은 근거에서 제외했습니다. 최종 부호화와 해석은 자격을 갖춘 전문가의 판단 영역입니다.
