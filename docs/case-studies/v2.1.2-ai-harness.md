# v2.1.2 AI Harness Case Study

This case study explains how the AI assistant layer in the Rorschach Structural Summary calculator was hardened in v2.1.2.

The goal was not to make the app look different. The goal was to make the AI behavior safer, more consistent, easier to test, and easier to explain.

## Product Context

The app includes two AI-assisted workflows:

- Coding helper: helps review possible Rorschach coding choices for a selected response row.
- Interpretation helper: helps reason about Structural Summary values and interpretation questions.

Both workflows use BYOK, meaning the user provides their own OpenAI API key. The key is not stored in the server database. It is held in an encrypted HttpOnly cookie for a limited session window.

This product deliberately keeps a human-in-the-loop boundary. The AI may suggest, explain, compare, or ask for missing observations. It should not silently score a protocol, finalize a diagnosis, or replace a trained clinician's judgment.

## Original Problem

Before v2.1.2, the AI layer was too informal.

Some parts were controlled by short prompt snippets. Some parts were controlled by output length assumptions. Some behavior was checked only by manual testing. That made the system feel like a simple LLM wrapper rather than a product-grade assistant layer.

The main risks were:

- The answer could stop early when the model hit an output limit.
- The coding helper could sound too certain when the observation was ambiguous.
- The interpretation helper could overreach from limited data.
- The assistant could imply it would edit or apply row values automatically.
- It was hard to prove that the reference knowledge layer was being used in the intended way.
- It was hard to show release evidence without exposing API keys or raw clinical-style model outputs.

## Design Decision

The fix was to introduce a small AI harness.

In this project, "harness" means the control layer around the model call. It is not the model itself. It is the code and policy layer that decides:

- which workflow profile is being used,
- what system instructions are attached,
- how long the response may be,
- what metadata is recorded,
- how streaming completion is interpreted,
- how failures and incomplete responses are detected,
- how the behavior is tested before release.

This made the AI layer more like an engineered subsystem and less like a raw prompt pasted into an API call.

The coding helper now has stronger instructions for:

- candidate and provisional language,
- clinician review,
- missing-observation questions,
- no automatic row editing,
- caution around FQ, Popular/P, Z, movement, and severe special scores.

The interpretation helper now has stronger instructions for:

- data anchors,
- provisional hypotheses,
- low-R validity caution,
- diagnostic and treatment boundaries,
- legal and forensic boundaries,
- no invented variables when the user has not supplied enough values.

## Evaluation Strategy

The eval work used two layers.

Static contract tests check that the harness, prompts, fixtures, and retrieval contracts are wired correctly.

Live eval runs real OpenAI calls against synthetic scenarios. These scenarios are not meant to replace expert clinical review. They are regression tests for product behavior: incomplete answers, unsafe certainty, missing HITL boundaries, forbidden overclaims, and weak reference-grounding.

The saved JSONL logs include:

- fixture ID,
- model ID,
- status,
- issue types,
- token usage,
- output character count,
- estimated cost.

The saved logs do not include:

- API keys,
- raw model answers,
- private clinical records,
- user secrets.

## What The Eval Found

The eval pass found several useful problems before release.

One evaluator rule was too brittle. It flagged safe answers that said the assistant could not make a legal judgment, simply because the phrase "legal judgment" appeared. The evaluator was changed to allow clearly negated safety-boundary statements while still rejecting unsafe affirmative claims.

The coding assistant sometimes gave a refusal without enough helpful next-step framing. The Korean direct-action boundary was strengthened so the assistant says it cannot apply row values automatically, but can help discuss candidate codes for clinician review.

Broad interpretation questions sometimes encouraged invented structure. The interpretation prompt was strengthened so that when no Structural Summary values are supplied, the assistant gives a review order instead of inventing findings.

One `gpt-5.5` failure was not a product failure. It was an eval cap problem. The live eval used a 700-token cap even though the product profile allowed more. The eval was corrected to use model-appropriate output caps.

## Security And Privacy Posture

The release kept the existing BYOK and privacy boundaries:

- API keys are not stored in the server database.
- OpenAI calls use `store: false`.
- Live eval commands loaded the key only into `OPENAI_API_KEY`.
- The environment variable was removed after each command.
- Public logs store metadata, not raw model output.
- Public source excludes local environment files, Vercel project settings, local logs, cache, `node_modules`, and private work notes.

## Remaining Limits

This release did not make the AI clinically authoritative.

The evals are regression and behavior tests. They check whether the assistant follows product boundaries. They do not prove clinical correctness in the way an expert review panel would.

The next useful improvement would be a human rubric:

- What counts as a good coding-helper answer?
- What counts as a safe interpretation-helper answer?
- When should the assistant ask for more data instead of answering?
- Which mistakes are minor wording issues, and which are release-blocking safety failures?

That rubric can become the foundation for future v2.1.x AI quality work.
